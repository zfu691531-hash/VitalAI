# Shared 模块开发手册

## 模块定位

`VitalAI/shared/` 是项目内部的轻量共享层。

它的存在目的不是承载大量逻辑，而是放那些：

- 真正跨模块复用
- 语义稳定
- 但又不适合放进 `Base/`

的少量对象。

## 当前结构

```text
VitalAI/shared/
├─ constants/
├─ errors/
├─ schemas/
└─ utils/
```

## 当前状态

当前 `shared/` 仍然保持很轻，基本还是骨架状态，这是健康的。

这说明目前项目还没有把 `shared/` 用成杂物间。

## 子目录职责

### `constants/`

适合放：

- 系统级常量
- 稳定的项目内枚举映射
- 少量跨层共享的固定字面值

不要放：

- 只在某个领域使用的常量
- 明显属于 `platform` 协议层的枚举

### `errors/`

适合放：

- 跨层复用的通用异常类型
- 可以从 interfaces 一路传到 application 的轻量错误定义

不要放：

- 某个领域专属的细碎异常全集

### `schemas/`

适合放：

- 跨层共享、又不属于 platform contract 的轻量数据结构

不要放：

- platform 的运行时协议
- domain 的核心 outcome

### `utils/`

适合放：

- 项目内部极少量共用的小工具

不要放：

- `Base/RicUtils` 已经提供的通用工具
- 一时不知道放哪的函数

## 模块要求

- 小
- 稳
- 克制

判断标准很简单：

如果一个对象明显属于下面任意一层，就不要放进 `shared/`：

- `platform`
- `application`
- `domains`
- `interfaces`
- `Base`

## 模块风格

- 只承载轻量对象
- 语义尽量稳定
- 命名要非常明确
- 新增前先问一句：是不是别的层更合适

## 交付标准

一个对象想进入 `shared/`，至少应满足：

1. 被多个模块稳定复用
2. 不明显隶属于某个单独领域
3. 放在 `Base` 又不合适
4. 放回某一层会造成明显重复

## 推进注意事项

### 1. 不要把 `shared/` 用成回收站

“暂时不知道放哪”绝不是进入 `shared/` 的理由。

### 2. 优先检查 `Base`

如果是通用工具、通用模型、配置辅助，先看 `Base` 是否已有现成能力。

### 3. 平台协议不要放进来

像消息信封、反馈事件、中断信号，这些都应该继续留在 `platform/`。

### 4. 领域结果不要放进来

像 `HealthTriageOutcome`、`DailyLifeSupportOutcome` 这类对象，应该继续留在各自 domain。

## 推荐使用方式

只有在出现下面这种情况时，再考虑引入 `shared/` 对象：

- `application`、`interfaces`、`domains` 三层里都在用同一个轻量结构
- 这个结构不是平台协议，也不是领域语义，只是项目内部稳定共享对象

## 当前最适合继续做的点

当前不建议主动往 `shared/` 里加东西。

更好的策略是：

- 先继续开发真实模块
- 当共享对象被真实证明存在时，再把它收进 `shared/`

## 常见错误

- 把“公共”误解成“都放 shared”
- 把本来应该在 `Base` 的工具复制进 `shared`
- 把某个领域暂时通用的对象错误抽成 shared
