# VitalAI 项目进展报告

报告日期：2026-04-16

## 一、项目概述

VitalAI 是一个面向**养老照护场景**的中心驱动型智能后端系统，构建在通用基础能力层 `Base/` 之上。

项目目标覆盖：
- 日常协助（daily_life）
- 健康监测与风险识别（health）
- 精神关怀与陪伴（mental_care）
- 面向家属的反馈与报告（reporting）
- 长期画像、记忆与关系演进（profile_memory）

项目性质：不是独立 greenfield 小项目，而是基于 `Base/` 的服务端业务系统，复用 Base 的通用能力，在 VitalAI 内承载业务编排、平台机制和领域逻辑。

## 二、架构设计

### 2.1 总体架构

VitalAI 采用**有清晰边界的 Modular Monolith** 架构，当前不做微服务拆分。

```
VitalAI/
├── application/    # 应用编排层：command / query / use case / workflow / assembly
├── platform/       # 平台与运行时层：跨领域共享机制
├── domains/        # 领域层：业务领域自身语义与规则
├── interfaces/     # 接口层：HTTP API / scheduler / consumer 等外部入口
├── shared/         # 共享层：少量跨层轻量稳定对象
└── main.py
```

上层依赖关系：
```
用户输入
→ interfaces 薄入口
→ application command / use case / workflow
→ intent recognition / routing
→ domains 领域能力
→ platform runtime / feedback / security / observability
→ 统一响应与可验证报告
```

### 2.2 各层职责

| 层 | 职责 | 关键原则 |
|---|---|---|
| **platform/** | 跨领域系统机制：messaging, feedback, arbitration, interrupt, runtime, observability, security | 保持组件拆分，不回退为单体 supervisor；协议优先 typed contract |
| **domains/** | 业务领域语义：health, daily_life, mental_care, reporting, profile_memory | 高聚合，领域对象命名清楚，outcome 优于散落返回值 |
| **application/** | 用例编排：command, query, use case, workflow, assembly | 编排层不承载领域判断和接口细节 |
| **interfaces/** | 外部入口适配：API, scheduler, consumer | 保持薄入口，不做业务判断 |
| **shared/** | 跨层轻量共享对象 | 小而稳，不变成杂物间 |
| **Base/** | 通用基础能力：Config, Ai, Client, Repository, Models, Service, RicUtils | VitalAI 不反向污染 Base |

### 2.3 Base 复用规则

- 通用基础能力优先复用 Base
- VitalAI 运行时语义与业务专属结构留在 VitalAI
- 新增模块前必须先检查 Base 是否已有能力
- 如果缺的是通用能力 → 补到 Base
- 如果缺的是 VitalAI 特有能力 → 放在 VitalAI

### 6.2 当前关键风险

1. **BERT bootstrap 模型不具备生产泛化能力**
   - 当前 108/108 holdout 通过依赖 BERT + hard-case guard + fallback + decomposition detector 的组合能力
   - 部分意图仍依赖 bert_low_confidence_fallback 成功路由
   - 没有真实用户语料前，不建议继续人工堆样本

2. **第二层意图拆分尚无真实 LLM backend**
   - 当前只有契约、placeholder、schema validator、adapter shell
   - 不会从拆分结果直接执行领域 workflow

3. **Runtime snapshot 持久化不是生产级方案**
   - 本地文件型方案仅适合开发环境
   - 缺少清理策略、迁移策略和分布式/高可用方案

4. **领域纵切深度仍然偏浅**
   - health / daily_life / mental_care 只有最小历史记录和只读查询闭环
   - 不是完整健康档案、提醒调度、医疗规则引擎、心理量表或多轮陪伴系统

5. **Admin 权限只有最小 token 基线**
   - 不是完整用户/角色权限体系

6. **依赖面过大（P7 仍开放）**
   - requirements.txt 包含大量当前应用未使用的依赖

7. **安全审查仍有误报风险（P1 部分解决）**
   - 嵌套 payload 的递归审查可能不完整
