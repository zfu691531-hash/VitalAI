# 前端开发需求文档

> 面向前端工程师的开发指引，基于项目整体架构拆分。开发前请先阅读项目根目录 `README.md` 了解全局结构。

## 一、技术栈与工具链

| 工具 | 版本要求 | 说明 |
|---|---|---|
| Vue3 | 3.4+ | 使用Composition API（`<script setup>`） |
| Vite | 5.x | 开发服务器 + 构建工具 |
| Element Plus | 2.x | UI组件库，按需导入 |
| Pinia | 2.x | 状态管理 |
| Vue Router | 4.x | 路由管理 |
| Axios | 1.x | HTTP请求 |
| xlsx | 0.18+ | 前端Excel导出 |
| html-to-image | 1.x | DOM转图片导出 |
| SCSS | — | CSS预处理器 |

## 二、目录结构与职责

```
frontend/src/
├── main.js              ← 你要写的第一个文件，注册所有插件
├── App.vue              ← 根组件
├── api/                 ← 【第一优先】封装所有后端接口调用，每个文件对应后端一个路由模块
├── stores/              ← Pinia状态管理
├── router/              ← 路由定义 + 权限守卫
├── composables/         ← 可复用逻辑（useTable/usePermission/useAiGenerate）
├── utils/               ← 工具函数
├── directives/          ← 自定义指令（v-permission）
├── styles/              ← 全局样式变量
├── layouts/             ← 页面布局骨架
├── views/               ← 【核心】22个页面，按模块组织
└── components/          ← 【重点】8个公共组件，减少重复代码
```

## 三、开发优先级与节奏

### 第一阶段：基础骨架（先跑通，再美化）

按以下顺序开发，每完成一步确保可运行：

```
1. 项目初始化
   ├── npm create vite@latest → 选择Vue模板
   ├── 安装依赖：element-plus, pinia, vue-router, axios, xlsx, html-to-image, sass
   └── vite.config.js 配置开发代理 /api → http://localhost:8000

2. 全局配置
   ├── src/main.js：注册ElementPlus(按需)、Pinia、Router、v-permission指令
   ├── src/api/request.js：Axios实例封装（核心！所有请求的入口）
   │   ├── 请求拦截器：从localStorage读取token → 注入Authorization头
   │   ├── 响应拦截器：code非200时ElMessage.error提示
   │   └── 401响应：清除token，跳转/login
   ├── src/stores/user.js：存储token、用户信息、角色
   └── src/utils/constants.js：定义角色枚举ROLES = { STUDENT:'student', TEACHER:'teacher', ADMIN:'admin' }

3. 路由与布局
   ├── src/router/routes.js：定义所有路由，每个route的meta标记requiredRoles
   ├── src/router/index.js：全局前置守卫（无token→登录页，无权限→403）
   ├── src/layouts/BlankLayout.vue：登录页使用
   └── src/layouts/MainLayout.vue：
       ├── 左侧栏：el-menu，菜单项根据用户角色动态过滤
       ├── 顶部栏：用户名 + 退出按钮
       └── 内容区：<router-view />

4. 公共组件（后续页面复用这些组件）
   ├── src/components/common/SearchBar.vue
   ├── src/components/common/Pagination.vue
   ├── src/components/common/ConfirmDialog.vue
   ├── src/components/common/FileUpload.vue
   └── src/components/common/ExportButton.vue
```

## 四、前后端接口对接规范

### 请求/响应格式

所有后端接口统一返回：
```json
{
  "code": 200,
  "msg": "操作成功",
  "data": { ... }
}
```
前端 `api/request.js` 的响应拦截器统一处理：`code !== 200` 时 `ElMessage.error(msg)`。

### API文件与后端路由的对应关系

| 前端文件 | 后端路由 | 基础路径 |
|---|---|---|
| `api/auth.js` | `api/auth.py` | `/api/auth` |
| `api/user.js` | `api/users.py` | `/api/users` |
| `api/student.js` | `api/students.py` | `/api/students` |
| `api/teacher.js` | `api/teachers.py` | `/api/teachers` |
| `api/class_.js` | `api/classes.py` | `/api/classes` |
| `api/score.js` | `api/scores.py` | `/api/scores` |
| `api/schoolRule.js` | `api/school_rules.py` | `/api/school-rules` |
| `api/aiTools.js` | `api/ai_tools.py` | `/api/ai` |
| `api/aiHistory.js` | `api/ai_history.py` | `/api/ai-history` |

### 后端接口文档地址

后端启动后访问 `http://localhost:8000/docs`（FastAPI自动生成的Swagger文档），以此为准进行前端对接。

## 五、权限控制实现方案

### 页面级权限（路由守卫）

```js
// router/routes.js 中每个路由的 meta
{
  path: '/students',
  component: () => import('@/views/student/StudentList.vue'),
  meta: { title: '学生管理', requiredRoles: ['teacher', 'admin'] }
}
```

```js
// router/index.js 全局前置守卫
router.beforeEach((to, from, next) => {
  if (to.meta.requiredRoles) {
    const userRole = useUserStore().role
    if (!to.meta.requiredRoles.includes(userRole)) {
      next('/403')  // 无权限跳转403页
      return
    }
  }
  next()
})
```

### 按钮级权限（v-permission指令）

```vue
<!-- 仅admin可见 -->
<el-button v-permission="['admin']">删除</el-button>

<!-- teacher和admin可见 -->
<el-button v-permission="['teacher', 'admin']">编辑</el-button>
```

### 各页面的权限矩阵

| 页面 | 学生 | 教师 | 管理员 |
|---|---|---|---|
| 首页 | 可见 | 可见 | 可见 |
| 学生管理 | 不可见 | 可见（本班） | 可见（全量） |
| 教师管理 | 不可见 | 仅查自己 | 可见（全量） |
| 班级管理 | 不可见 | 可见（绑定班级） | 可见（全量） |
| 成绩管理 | 可见（仅自己） | 可见（本班） | 可见（全量） |
| 校规管理 | 不可见 | 不可见 | 可见 |
| 评语/话术/润色/诊断/策划 | 不可见 | 可见 | 可见 |
| 校规问答 | 可见 | 可见 | 可见 |
| 模拟面试 | 可见 | 可见 | 可见 |
| 分组分班-教师端 | 不可见 | 可见（本班） | 不可见 |
| 分组分班-校务端 | 不可见 | 不可见 | 可见 |

## 六、交互规范（必须遵守）

### 通用交互

| 场景 | 要求 |
|---|---|
| 所有增/删/改操作 | 弹出二次确认弹窗（ConfirmDialog），明确提示操作内容 |
| 表单提交 | 前端必填项校验，未填写时提示"请补充" |
| 操作反馈 | 成功ElMessage.success，失败ElMessage.error |
| 表格 | 分页展示，支持关键词搜索 + 多条件筛选 |
| 批量操作 | 支持批量选择、批量导入导出 |

### AI工具交互

| 场景 | 要求 |
|---|---|
| 点击生成按钮 | loading动画 + 文字"正在生成，请稍候" |
| 生成失败 | 明确错误提示 + 重试按钮 |
| 生成成功 | 一键复制 + 导出按钮 + 自动保存历史 |
| 多轮对话 | 对话气泡，AI回复时显示loading |

## 八、composables复用指南

### useTable.js（所有列表页共用）

```js
const { pagination, loading, tableData, fetchData, handleSearch, handlePageChange } = useTable(fetchApi)
```

### usePermission.js（权限判断）

```js
const { hasRole, canAccess } = usePermission()
if (hasRole('admin')) { /* ... */ }
```

### useAiGenerate.js（所有AI工具页面共用）

```js
const { loading, error, result, generate, retry } = useAiGenerate(apiFn)
// generate(params) → 自动处理loading/error/result
```

## 九、代码质量红线

1. **禁止**在组件中直接写后端地址，所有请求走 `api/` 层
2. **禁止**在组件中直接操作localStorage，走 `stores/user.js`
3. **禁止**重复写表格分页/搜索逻辑，使用 `useTable` composable
4. **禁止**重复写AI生成loading/retry逻辑，使用 `useAiGenerate` composable
5. 所有el-table的列定义、操作按钮位置在各CRUD页面中保持一致
6. 所有表单弹窗统一使用el-dialog，统一确认/取消按钮位置
