# VitalAI 技术文档解读

---
> 为什么取这个名字作为你的项目名称？

### 回答要点

#### 一、词源拆解

**Vital** 来自拉丁语 *vitalis*，词根是 *vita*（生命），有三层含义：
- 生命本质：与生命相关、维持生命所必需的
- 关键核心：至关重要的、不可或缺的
- 生机活力：充满活力的、朝气蓬勃的

**AI** = Artificial Intelligence（人工智能）

---

#### 二、三层命名寓意

**第一层：业务场景直指——"守护生命"**

Vital = 生命 + 关键，AI = 人工智能

> **VitalAI = 用AI守护生命中至关重要的部分**

项目聚焦老人陪护场景，老人最核心的需求——健康、安全、陪伴、尊严，这些是生命中最Vital的部分。项目从第一天起就不是做"智能音箱"或"聊天机器人"，而是做真正触及生命本质的陪护系统——监测跌倒、提醒服药、识别情绪异常、保护资金安全，这些功能关乎生命安危。

**第二层：技术哲学映射——"关键路径"**

在系统设计层面，Vital还有另一层含义：系统中至关重要的核心能力。

项目的四层基础机制——通信层、反馈层、仲裁层、中断层，是系统能力中的Vital要素。没有通信Agent就是孤岛，没有反馈系统就无法进化，没有仲裁多Agent就会冲突混乱，没有中断紧急情况无法优先处理。项目的核心价值不是某个具体功能，而是构建了一个能够可靠运行的"生命系统"——能自我调节、自我保护、自我恢复。

**第三层：产品愿景承载——"让生命更有活力"**

Vital = 使...充满生机

项目的终极愿景不是让老人活着，而是让老人的晚年生活有质量、有尊严、有活力。

对比传统养老方案的被动响应、功能孤岛、冷冰冰监控、短期应急处理，VitalAI追求主动预判、统一调度、有温度陪伴、长期记忆学习。通过AI的长期记忆能力，记住老人的偏好、习惯、故事，让陪伴真正有温度。

---

#### 三、与竞品的差异化定位

市面上类似产品的命名问题：
- 硬件类（智能音箱、陪伴机器人）：强调产品形态，弱化核心价值
- 平台类（智慧养老平台、康养云）：官方感太强，缺乏温度
- 功能类（监护系统、跌倒检测仪）：功能导向，缺乏愿景感

VitalAI的差异化：
- 不强调形态（不是硬件、不是平台）
- 不强调功能（不是监护、不是检测）
- 强调价值（生命的关键、让生命有活力）

这个名字可以容纳硬件接入、软件服务、平台能力、未来医疗健康扩展。

---

#### 四、技术实现的呼应

代码架构也在呼应这个名字。Vital的系统组件——DecisionCore（决策核心/大脑）、HealthMonitor（健康监控/心跳）、Interrupt（紧急中断/免疫系统）、Failover（故障恢复/自愈能力）、Degradation（降级运行/生存本能），这些就是一个有生命力的系统应该具备的能力。一个没有中断、没有恢复、没有降级的系统，是脆弱的，而不是Vital的。

---

#### 五、品牌与传播考量

**国际化友好**：Vital和AI都是国际通用词，英语母语者秒懂，非英语母语者也能理解，无文化禁忌。

**记忆成本低**：VitalAI两个音节，朗朗上口。对比"IntelligentElderlyCompanionSystem"太长、"智慧养老综合服务平台"官方腔、"VitalAI"简洁有力。

---

#### 六、总结

VitalAI这个名字，既是对业务场景的直白表达（守护生命的关键），也是对技术哲学的隐喻（构建有生命力的系统），更是对产品愿景的承载（让生命更有活力）。

它告诉用户：这不是一个冰冷的监控工具，而是一个真正关心你生命的AI伙伴。

相比CareAI强调单向"照料"、HealthAI范围太窄，VitalAI强调"生命的关键"，既包含健康，也包含陪伴、尊严、安全，更包含让生命有活力的愿景。

## 知识点 1：VitalAI 到底是个什么东西？

**VitalAI 是一个"养老智能大脑"——一个专门为老人陪护场景设计的后端 AI 系统。**

你可以想象一个这样的画面：

```
┌──────────────────────────────────────────────────┐
│                                                  │
│              👴 老人                              │
│           "我头晕"                                │
│              │                                   │
│              ▼                                   │
│     ┌─────────────────┐                          │
│     │   VitalAI 大脑   │  ← 这就是你的项目         │
│     │                 │                          │
│     │  听懂了 → 健康预警 │                          │
│     │  记住了 → 长期记忆 │                          │
│     │  发现复杂 → 拆分   │                          │
│     │  出事了 → 紧急中断 │                          │
│     └─────────────────┘                          │
│                                                  │
└──────────────────────────────────────────────────┘
```

但它**不是**一个聊天机器人，也不是一个 APP。它是一个**纯后端服务**，相当于整个养老智能系统的"后脑勺"——负责理解、记忆、决策，但不负责界面展示。

面试的时候，你可以这样一句话概括：

> "VitalAI 是一个面向养老陪护场景的服务端多 Agent 系统，核心能力是统一调度、长期记忆、意图识别、冲突仲裁和紧急中断。"

## 知识点 2：仓库为什么要分成"两层"？

你打开项目，第一眼看到的就是根目录下两个大文件夹：

```
d:/Users/Windows/PycharmProjects/VitalAI/
│
├── Base/          ← 地基（通用技术能力）
│
├── VitalAI/       ← 房子（业务系统本身）
│
└── main.py        ← 启动入口（就一行代码，转发到 VitalAI）
```

为什么要这样拆？我用一个比喻：

```
🏗️ 盖楼

Base/  =  地基 + 水管 + 电线 + 电梯
         → 这些东西，不管你盖住宅楼、写字楼、商场，都要用
         → 它是"通用的"

VitalAI/ =  这栋楼本身的设计
         → 养老公寓的布局、呼叫按钮、无障碍通道
         → 它是"业务专属的"
```

**关键原则：Base 只放通用能力，VitalAI 的业务逻辑绝不塞回 Base。**

为什么？因为 Base 以后可能被其他项目复用。如果把养老业务逻辑混进去，下次做个"儿童教育 AI"，就得带着养老代码，那就乱了。

具体来说：

| 层 | 解决什么 | 举例 |
|---|---|---|
| `Base/` | 怎么连数据库、怎么调 LLM、怎么配日志 | SQLite 连接、Qwen/DeepSeek 调用、日志初始化 |
| `VitalAI/` | 怎么理解老人意图、怎么调度健康预警 | 意图识别、健康分诊、快照恢复 |

面试的时候可以这样表述：

> "项目采用 Base + 业务层分离的架构，Base 作为可复用的技术基础设施，提供 DB/LLM/日志等通用能力；VitalAI 专注于养老陪护的业务逻辑，两者边界清晰，Base 不承载任何业务语义。"

**但 VitalAI 实际用到的 Base 能力，只有这 5 个：**


打个比方：Base 是一个大超市，VitalAI 是一个家庭，目前只从超市买了 5 样东西回家用。其他货架上还有东西，但暂时没用到。

## 知识点 3：VitalAI 内部的五层架构

VitalAI 里面不是平铺一堆文件，而是分了五层，像一个洋葱一样一层包一层：

```
        从外到内 ↓

┌──────────────────────────────────────────┐
│  ⑤ interfaces/  接口层                    │  最外层，最薄
│  "外面怎么进来"                            │
│  ┌──────────────────────────────────────┐│
│  │  ④ application/  应用层               ││  编排中心
│  │  "先做什么再做什么"                     ││
│  │  ┌──────────────────────────────────┐││
│  │  │  ③ platform/  平台层              │││  系统机制
│  │  │  "系统怎么运转"                    │││
│  │  │  ┌──────────────────────────────┐│││
│  │  │  │  ② domains/  领域层           ││││  最核心
│  │  │  │  "业务规则是什么"               ││││
│  │  │  └──────────────────────────────┘│││
│  │  └──────────────────────────────────┘││
│  └──────────────────────────────────────┘│
│                                          │
│  ① shared/  共享层（横切，少量通用常量/错误） │
└──────────────────────────────────────────┘
```

每一层只干一件事：

| 层 | 一句话职责 | 打比方 |
|---|---|---|
| **interfaces** | 外部怎么进入系统 | 酒店大堂，客人从这进来 |
| **application** | 编排完整业务流程 | 酒店经理，安排客人入住的每一步 |
| **platform** | 系统级通用机制 | 酒店的保安、监控、消防系统 |
| **domains** | 纯业务逻辑 | 酒店的各个服务部门（餐饮/住宿/SPA） |
| **shared** | 少量通用常量和错误 | 酒店的公共标识牌 |

**依赖方向只有一条：外层可以调用内层，内层不能调用外层。**

```
interfaces → application → platform
                         → domains
shared ← 所有人都能用，但它很小很精
```

这意味着 `domains`（领域层）是最纯粹的——它只关心业务规则，不知道 HTTP 是什么，不知道数据库是什么，不知道消息队列是什么。

面试时这样说：

> "VitalAI 采用洋葱架构，五层分层：接口层、应用层、平台层、领域层、共享层。依赖方向严格从外到内，领域层最纯粹，不依赖任何外部技术细节。"

### 为什么要分五层？——不分的后果

大多数小项目是这样组织的：

```
❌ 传统平铺式（按技术角色分）

project/
├── models/       所有数据模型放一起
├── services/     所有业务逻辑放一起
├── routers/      所有接口放一起
└── utils/        工具函数
```

短期看没问题，但项目一大就会出现三个致命问题：

**问题1：业务逻辑散落**

```
假设你要改"健康预警"功能：

你需要同时改：
  models/health.py      ← 数据模型在这
  services/health.py    ← 业务逻辑在这
  routers/health.py     ← 接口在这
  services/alert.py     ← 分诊逻辑跑到另一个文件了
  utils/notify.py       ← 通知逻辑又跑这了

改一个功能，跳5个目录 😵
```

**问题2：系统机制到处复制**

```
如果没有 platform/ 层：

health/   有一套冲突处理逻辑
daily_life/ 也有一套冲突处理逻辑
mental_care/ 又写了一套中断逻辑

三份代码，三个 bug，改一处漏两处 😵
```

**问题3：接口层越来越胖**

```
如果没有 application/ 层：

routers/health.py 里有：
  - 参数校验
  - 业务编排（先查库、再调服务、再写反馈）
  - 数据库读写
  - 业务决策

接口层变成"什么都有"，换接入方式就全废了 😵
```

**五层架构怎么解决这些问题：**

```
对比：

❌ 平铺式：按技术角色分，一个功能的代码散在5个目录
✅ 洋葱式：按业务领域聚，一个功能的代码就在一个地方

❌ 平铺式：系统机制（冲突/中断/反馈）各模块自己写自己的
✅ 洋葱式：platform/ 统一提供，所有领域复用

❌ 平铺式：接口层又做路由又做业务又做存储
✅ 洋葱式：接口层只做"接收请求→调用应用层→返回结果"，极薄
```

### 与其他架构的对比

面试时可能被问到的几种常见后端架构：

```
┌──────────────┬──────────────────┬──────────────────┬──────────────────┐
│              │   传统 MVC        │   DDD 分层        │   本项目洋葱架构   │
├──────────────┼──────────────────┼──────────────────┼──────────────────┤
│ 组织方式      │ 按技术角色分       │ 按领域分          │ 按领域分+横切平台  │
│              │ models/services   │ 每个领域一个包     │ 每个领域一个包     │
│              │ routers/utils     │                  │ + 独立平台层       │
├──────────────┼──────────────────┼──────────────────┼──────────────────┤
│ 系统机制      │ 散在各处           │ 散在各处          │ platform/ 统一    │
│ (冲突/中断)   │ 各写各的           │ 各写各的          │ 一套复用          │
├──────────────┼──────────────────┼──────────────────┼──────────────────┤
│ 接口层厚度    │ 胖（含业务逻辑）    │ 中等              │ 极薄              │
├──────────────┼──────────────────┼──────────────────┼──────────────────┤
│ 适合规模      │ 小项目             │ 中大项目          │ 大型多Agent系统    │
├──────────────┼──────────────────┼──────────────────┼──────────────────┤
│ 多Agent协作   │ 不支持             │ 需自己补          │ 原生支持           │
├──────────────┼──────────────────┼──────────────────┼──────────────────┤
│ 运行时韧性    │ 无                │ 无               │ 快照/故障转移/降级  │
│ (快照/故障转移)│                  │                  │ 开箱即用           │
└──────────────┴──────────────────┴──────────────────┴──────────────────┘
```

**最关键的区别是第三列：本项目的 platform/ 层。**

普通 DDD 项目只有领域层，没有专门的"平台层"。但你的项目是一个**多 Agent 协作系统**，天然需要：
- Agent 之间怎么通信 → messaging
- 冲突了怎么办 → arbitration
- 出事了怎么打断 → interrupt
- 挂了怎么恢复 → runtime（快照/故障转移）

这些不是某个业务的逻辑，而是**整个系统共享的机制**。如果不抽成 platform/，就会每个领域自己写一份。

面试的时候这样说：

> "相比传统 MVC 按技术角色平铺，我们的洋葱架构按领域聚合，改一个功能只动一个目录。相比普通 DDD，我们多了一层 platform，专门承载多 Agent 系统的通信、仲裁、中断、快照等系统级机制，避免每个领域重复实现。"

## 知识点 4：领域层 —— 业务核心长什么样

领域层是整个系统最纯粹的部分，只关心"业务规则是什么"。

当前有 5 个领域：

```
domains/
├── health/          🏥 健康监测
├── daily_life/      🍚 日常生活
├── mental_care/     💚 心理关怀
├── profile_memory/  🧠 用户画像记忆
└── reporting/       📊 报告反馈
```

**每个领域内部，结构完全一致**，像一个"标准间"：

```
domains/health/           ← 拿 health 举例
│
├── agents/      🤖 领域 Agent（当前占位，未来放智能体逻辑）
├── models/      📦 领域模型（实体 + 快照）
├── policies/    📋 领域策略（阈值、规则）
├── repositories/💾 仓储（负责持久化，底层用 Base 的 SQLite）
└── services/    ⚙️  核心业务服务（这个领域真正干活的逻辑）
```

用生活比喻：

```
把每个领域想象成一个"部门"：

agents/    = 部门里的专员（未来会变聪明，现在还是实习生）
models/    = 部门的档案格式（记录长什么样）
policies/  = 部门的规章制度（什么情况该怎么判）
repositories/ = 部门的档案柜（数据存取）
services/  = 部门的核心业务能力（真正干活的）
```

**四个已落地的领域，各自的最小闭环：**

```
🏥 health（健康）
   写入：老人说"我头晕" → 生成健康预警（risk_level: high/medium/low）
   读取：查询某老人最近的预警历史

🍚 daily_life（日常生活）
   写入：老人需要帮助 → 生成签到（need: meal_support/medication/...）
   读取：查询某老人最近的生活签到历史

💚 mental_care（心理关怀）
   写入：老人情绪信号 → 生成签到（mood_signal: calm/anxious/...）
   读取：查询某老人最近的心理关怀历史

🧠 profile_memory（画像记忆）
   写入：记住"老人喜欢喝茉莉花茶"（key=value 形式）
   读取：查询某老人的画像快照，或按 key 精确读取
```

每个领域都走的是 **CQRS 模式**——写操作和读操作完全分开：

```
                    ┌── Command（写）──→ Flow Workflow ──→ 写入 SQLite
                    │
用户请求 ──→ 应用层 ─┤
                    │
                    └── Query（读）───→ Query Workflow ──→ 从 SQLite 读取
```

面试时这样说：

> "领域层按业务聚合，每个领域统一包含 agents/models/policies/repositories/services 五个子目录。当前四个核心领域都已落地最小读写闭环，采用 CQRS 模式，写走 Command + FlowWorkflow，读走 Query + QueryWorkflow。"

---

## 知识点 5：平台层 —— 这个项目最精妙的设计

如果说领域层是"各个部门"，那平台层就是**"整栋楼的基础设施"**——电梯、消防、监控、保安，所有部门共享，谁都不用自己建。

平台层有 **6 大子系统 + 1 个运行时核心**：

```
platform/
│
├── messaging/      📨 通信层    —— Agent 之间怎么说话
├── feedback/       🔄 反馈层    —— 做完事怎么反馈
├── arbitration/    ⚖️ 仲裁层    —— 冲突了听谁的
├── interrupt/      🚨 中断层    —— 出事了怎么打断
├── observability/  👁️ 可观测层  —— 系统在干什么看得见
├── security/       🔒 安全层    —— 敏感数据怎么保护
│
└── runtime/        🧠 运行时    —— 整个系统的"心脏"
    ├── decision_core.py         决策核心
    ├── event_aggregator.py      事件聚合器
    ├── snapshots.py             快照系统
    ├── failover.py              故障转移
    ├── health_monitor.py        健康监控
    ├── shadow_decision_core.py  影子决策核心（热备）
    ├── degradation.py           降级策略
    └── signal_wiring.py         信号桥接
```

核心概念：**为什么不是一个大 Supervisor？**

```
❌ 传统：一个大 Supervisor 管一切

┌─────────────────────────┐
│                         │
│      Supervisor         │
│                         │
│  接收事件               │
│  分析事件               │
│  做决策                 │
│  发指令                 │
│  监控心跳               │
│  保存快照               │
│  故障恢复               │
│  记日志                 │
│  安全审查               │
│                         │
│  （一个对象干十件事 😵）   │
│                         │
└─────────────────────────┘

问题：
  - 原始事件全量涌入，中心容易过载
  - 高频聚合和高价值决策抢同一资源
  - 日志 IO 拖慢决策
  - 心跳、调度、决策全耦合
  - 中心挂了，整体失能
```

```
✅ 本项目：拆成一组协作组件

┌────────────┐
│ 事件聚合器  │ ← 只负责：去重、合并、排序、TTL校验
└──────┬─────┘
       │ 结构化摘要（不是原始事件）
       ▼
┌────────────┐
│  决策核心   │ ← 只负责：查 handler → 执行 → 返回出站消息
└──────┬─────┘    （极简，不干杂活）
       │
  ┌────┼────┐
  ▼    ▼    ▼
快照  故障  降级
系统  转移  策略
       │
  ┌────┴────┐
  ▼         ▼
可观测     安全
记录       审查
```

**核心思想：中心仍然存在，但中心只保留最核心的决策能力，其他职责拆给专门组件。**

打个比方：

```
❌ 传统 Supervisor = 一个校长又要教书、又要管后勤、又要当保安、又要修水电

✅ 本项目 = 校长只做决策，教导主任管纪律，后勤主任管设施，保安管安全
           校长挂了，副校长（影子核心）立刻接上
```

面试时这样说：

> "我们没有采用单体 Supervisor 模式，而是把运行时拆成事件聚合器、决策核心、快照系统、故障转移、降级策略等独立组件。决策核心只做最轻量的分发，聚合器预处理原始事件，影子核心提供热备。这样避免了单点过载和单点故障。"

---

## 知识点 6：消息信封 —— 所有组件通信的"通用语言"

在讲运行时组件怎么协作之前，你得先理解一个东西：**MessageEnvelope（消息信封）**。

它是整个系统里**所有组件之间通信的唯一格式**，就像邮局的信封一样：

```
┌─────────────────────────────────────────┐
│           MessageEnvelope               │
│                                         │
│  from_agent: "health_agent"    ← 谁发的  │
│  to_agent:   "decision_core"   ← 发给谁  │
│  payload:   { risk_level: "high" } ← 内容│
│                                         │
│  priority:  CRITICAL          ← 多紧急   │
│  ttl:       60                ← 活多久   │
│  require_ack: true            ← 要不要回执│
│  expire_at: 2026-04-16 10:05 ← 何时过期  │
│                                         │
│  msg_id:    "uuid-xxx"        ← 消息ID   │
│  trace_id:  "trace-xxx"       ← 链路追踪 │
│  version:   1                 ← 版本号   │
└─────────────────────────────────────────┘
```

为什么要统一信封？

```
❌ 没有统一信封：

health 模块发消息：{ "type": "alert", "level": 3, "data": {...} }
daily_life 发消息：{ "action": "checkin", "urgency": "high" }
mental_care 发消息：{ "event": "mood", "signal": "anxious" }

每个模块格式不一样，决策核心要写三套解析逻辑 😵
```

```
✅ 有统一信封：

所有模块都用 MessageEnvelope，只是 payload 内容不同

决策核心只需要：拆信封 → 读 priority → 按 event_type 分发

一套解析逻辑，所有模块通用 ✅
```

**优先级有四级：**

```
CRITICAL  ──→ 最高，立即处理（如：老人摔倒）
HIGH      ──→ 紧急，尽快处理（如：血压异常）
NORMAL    ──→ 普通，正常排队（如：日常签到）
LOW       ──→ 最低，有空再处理（如：定期报告）
```

**TTL（生存时间）：** 消息超过这个时间还没处理就作废，防止过期消息干扰决策。

**require_ack：** 发送方要求接收方必须回复确认，防止消息丢失。

面试时这样说：

> "系统内所有组件间通信统一使用 MessageEnvelope，包含来源、目标、载荷、优先级、TTL、ACK 要求和链路追踪。这保证了任何组件发消息格式一致，运行时组件只需处理标准信封。"

---

## 知识点 7：运行时心脏 —— 一条事件的完整旅程

现在你知道了消息信封是通用通信格式，接下来看一条事件是怎么在运行时里走完一圈的：

```
假设：老人说"我头晕"，系统识别为健康预警

                    MessageEnvelope
                    (priority=CRITICAL)
                         │
                         ▼
              ┌─────────────────────┐
              │   事件聚合器         │
              │   EventAggregator   │
              │                     │
              │   做了什么？          │
              │   ✓ 过期消息扔掉      │
              │   ✓ 相同事件去重      │
              │   ✓ 按优先级排序      │
              │   ✓ 生成结构化摘要    │
              └──────────┬──────────┘
                         │
                    EventSummary
                   （干净、结构化的）
                         │
              ┌──────────┴──────────┐
              ▼                     ▼
     ┌──────────────┐      ┌──────────────┐
     │  信号桥接     │      │  决策核心     │
     │  SignalBridge │      │  DecisionCore│
     │              │      │              │
     │  同时推入：   │      │  查找 handler │
     │  👁️ 可观测记录 │      │  执行 handler │
     │  🔒 安全审查  │      │  返回出站消息  │
     └──────────────┘      └──────┬───────┘
                                  │
                                  ▼
                        ┌──────────────────┐
                        │  领域服务         │
                        │  (如健康分诊)     │
                        │                  │
                        │  产出三样东西：    │
                        │  📨 决策消息       │
                        │  🔄 反馈事件       │
                        │  ⚖️ 仲裁意图       │
                        └────────┬─────────┘
                                 │
                    ┌────────────┼────────────┐
                    ▼            ▼            ▼
              如果是关键时刻   如果中心挂了    如果压力太大
              ┌─────────┐  ┌──────────┐  ┌──────────┐
              │ 拍快照    │  │ 故障转移  │  │ 降级运行  │
              │ Snapshot │  │ Failover │  │Degradation│
              └─────────┘  └──────────┘  └──────────┘
```

关键理解点：

**1. 事件聚合器是"过滤器"**

```
原始输入                    聚合器输出
（乱、多、可能重复）          （干净、结构化）

"头晕" × 3 条               → 1 条 health_alert
1 条过期消息                 → 扔掉
2 条低优先级 + 1 条 CRITICAL  → CRITICAL 排最前面
```

**2. 决策核心是"极简分发器"**

```
它不思考，只分发：

收到 EventSummary
  → 查表：event_type="health_alert" → handler=health_triage
  → 调用 health_triage(summary)
  → 返回结果

就这么简单，不干杂活。
```

**3. 信号桥接是"横切面"**

```
每个关键动作，都同时做两件事：

  可观测记录  → 事后能查"当时发生了什么"
  安全审查    → 敏感数据自动扫描脱敏

不用每个组件自己写日志和审查，桥接自动做。
```

面试时这样说：

> "运行时的核心流程是：原始事件进入聚合器做去重排序，生成结构化摘要；摘要同时推入决策核心和信号桥接；决策核心按 event_type 分发到领域 handler；信号桥接统一做可观测记录和安全审查。每个组件职责单一，互不耦合。"

---

## 知识点 8：快照 + 故障转移 —— 系统挂了怎么办

养老系统有一个特殊要求：**不能随便挂**。如果老人正在发健康预警，系统崩了，这可不是小事。

所以系统设计了三道保险：

### 第一道：快照系统 —— 关键时刻"拍照存档"

```
就像打游戏时的"存档"：

关键时刻自动存档：
  ✓ 收到 CRITICAL 级别的健康预警 → 拍快照
  ✓ 高紧急度的日常签到          → 拍快照

快照里存了什么？
  ┌─────────────────────┐
  │  RuntimeSnapshot     │
  │                     │
  │  snapshot_id        │ ← 哪个场景的快照
  │  payload            │ ← 当时系统状态（决策数据）
  │  version            │ ← 版本号（会递增）
  │  trace_id           │ ← 哪次请求触发的
  │  captured_at        │ ← 什么时候拍的
  └─────────────────────┘

存哪？
  默认 → 进程内存（重启就没了）
  配置 VITALAI_RUNTIME_SNAPSHOT_STORE_PATH → 本地文件（重启还在）
```

### 第二道：故障转移 —— 主挂了，备立刻接上

```
┌────────────┐         ┌──────────────────┐
│            │  正常时   │                  │
│  主决策核心 │ ────────→│  影子决策核心     │
│  (active)  │  同步状态 │  (standby 热备)   │
│            │         │                  │
└─────┬──────┘         └────────┬─────────┘
      │                         │
      │ 主挂了！                 │
      ▼                         │
  ┌────────┐                    │
  │ 中断信号 │ ──────────────────┘
  │ TAKEOVER│  影子核心从最新快照恢复
  └────────┘  → 接管成为新的 active
```

```
打个比方：

主决策核心 = 主刀医生
影子决策核心 = 旁边的副手，一直在看主刀操作

主刀医生突然不行了
  → 护士发出中断信号
  → 副手根据刚才看到的进度（快照）立刻接上
  → 手术不中断
```

### 第三道：降级策略 —— 中心全挂了，系统还能保底

```
万一主和备都挂了呢？

┌─────────────────────────────────────────┐
│           Degradation 降级策略            │
│                                         │
│  Level 0：正常       一切正常运行         │
│  Level 1：自治惯性    中心不可用，         │
│                     各领域按最近规则自己跑  │
│  Level 2：保底运行    只处理 CRITICAL，    │
│                     其他全拒绝            │
│  Level 3：只记录      什么都执行不了，     │
│                     至少把事件记下来       │
└─────────────────────────────────────────┘
```

**三道保险的完整画面：**

```
正常情况：
  事件 → 聚合 → 决策核心处理 → 正常运行
                          ↓
                     关键时刻拍快照

出了小问题：
  事件 → 聚合 → 决策核心挂了 → 中断信号
                                ↓
                           影子核心接上
                           （从快照恢复状态）
                           → 继续运行

出了大问题：
  主和备都挂了 → 进入降级模式
                  → 至少保住 CRITICAL 事件
                  → 至少记录下来
```

面试时这样说：

> "运行时韧性靠三道保险：第一道是快照系统，关键时刻自动存档；第二道是故障转移，影子核心热备，主节点挂了立刻从快照恢复接管；第三道是降级策略，从正常到自治惯性到保底运行到只记录，逐步降级，至少保证 CRITICAL 事件不丢。"

---

## 知识点 9：中断 + 仲裁 —— 出事怎么打断，冲突怎么裁决

这两个机制是普通后端项目几乎不会有的，但在养老场景里至关重要。

### 中断层 —— "着火了，所有会议暂停！"

```
正常情况下，系统按优先级排队处理事件：

  队列：[日常签到] [记忆更新] [健康预警] [日常签到]
                                         ↑
                                      优先处理

但如果突然来了一个 P0 紧急事件（老人摔倒）：

  ┌─────────────────────────────────────────┐
  │         InterruptSignal                  │
  │                                         │
  │  action: TAKEOVER / PAUSE / CANCEL      │
  │  priority: P0 / P1 / P2 / P3           │
  │  reason: "老人摔倒，需紧急处理"            │
  │  source: "health_monitor"               │
  └─────────────────────────────────────────┘

中断信号一来：
  ✗ 当前正在处理的低优先级任务 → 暂停或取消
  ✓ 紧急事件 → 立刻插队到最前面
  ✓ 触发快照 → 保存当前状态
  ✓ 通知故障转移 → 准备影子核心
```

```
打个比方：

医院里：
  普通门诊在排队看病 → 突然送来一个车祸重伤员
  → 护士发出中断信号
  → 普通门诊暂停
  → 急诊室立刻接手重伤员
  → 处理完再恢复门诊
```

**中断优先级：**

```
P0  🚨🚨🚨  生死攸关（老人摔倒、心脏异常）  → 立刻接管一切
P1  🚨🚨    紧急（血压异常持续升高）           → 暂停低优先级
P2  🚨      需要关注（情绪持续低落）           → 排队靠前
P3  📋      提醒级别（该吃药了）               → 正常排队
```

### 仲裁层 —— "两个 Agent 冲突了，听谁的？"

养老场景会出现这样的情况：

```
场景：老人刚做完手术

  健康Agent 说："老人需要卧床休息，不要外出"
  日常Agent 说："今天的购物任务还没完成，需要出门"

  两个 Agent 目标冲突！听谁的？
```

这时候就需要仲裁：

```
┌─────────────────────────────────────────────┐
│             IntentDeclaration                │
│          （每个 Agent 声明自己的意图）          │
│                                             │
│  agent_id: "health_agent"                   │
│  goal_type: SAFETY          ← 目标类型：安全  │
│  description: "术后卧床休息"                  │
│  priority: CRITICAL                         │
│  resources_needed: ["老人注意力"]              │
│                                             │
│  vs                                         │
│                                             │
│  agent_id: "daily_life_agent"               │
│  goal_type: DAILY_ROUTINE   ← 目标类型：日常  │
│  description: "完成购物任务"                   │
│  priority: NORMAL                           │
│  resources_needed: ["老人注意力"]              │
└─────────────────────────────────────────────┘

仲裁规则很简单：
  SAFETY + CRITICAL  vs  DAILY_ROUTINE + NORMAL
  → 安全优先，CRITICAL 优先
  → 健康Agent 赢
  → 购物任务推迟
```

**目标类型的优先级：**

```
SAFETY       🛡️ 安全      → 最高，生命安全大于一切
HEALTH       🏥 健康      → 次高，健康管理优先
DAILY_ROUTINE 🍚 日常     → 普通，日常事务
COMFORT      😊 舒适      → 最低，锦上添花
```

**中断 + 仲裁合在一起的画面：**

```
正常：各领域按优先级排队处理

出事了（P0 中断）：
  → 所有普通任务暂停
  → 紧急事件插队处理

处理中冲突了（两个 Agent 争资源）：
  → 各自声明意图
  → 仲裁器按目标类型 + 优先级裁决
  → 输的一方等或降级

紧急处理完了：
  → 恢复暂停的任务
```

面试时这样说：

> "中断层支持 P0-P3 四级优先级，P0 事件可以直接打断一切正在执行的低优先级任务。仲裁层让冲突的 Agent 各自声明意图和目标类型，按 SAFETY > HEALTH > DAILY > COMFORT 的优先级裁决。这两个机制保证紧急事件永远优先，冲突时有明确裁决规则。"

---

## 知识点 10：应用层 —— 编排中心，"谁先上谁后上"

领域层是"各个部门"，平台层是"基础设施"，但谁来**安排流程**？比如：

- 老人发来一条消息 → 先识别意图 → 再路由到对应领域 → 再写反馈 → 再生成报告
- 这个顺序谁定？ → **应用层**

应用层里有四个关键概念：

```
application/
│
├── commands/     📝 命令（写操作的输入）
├── queries/      🔍 查询（读操作的输入）
├── use_cases/    🎯 用例（一个步骤的业务逻辑）
└── workflows/    🔗 工作流（多个步骤串起来）
```

用看病的流程来理解：

```
┌──────────────────────────────────────────────────┐
│                                                  │
│  Command = 挂号单                                │
│  "我叫张三，我要看健康科"                           │
│                                                  │
│  Query = 查询单                                  │
│  "查一下我上次的检查记录"                           │
│                                                  │
│  UseCase = 一个检查步骤                           │
│  量血压是一个用例、抽血是一个用例、拍片是一个用例     │
│                                                  │
│  Workflow = 完整看病流程                          │
│  挂号 → 量血压 → 抽血 → 拍片 → 诊断 → 开药        │
│  （把多个用例按顺序串起来）                         │
│                                                  │
└──────────────────────────────────────────────────┘
```

**以健康预警为例，完整的编排过程：**

```
用户说"我头晕"
       │
       ▼
  ┌──────────┐
  │ Command   │  HealthAlertCommand
  │ 命令      │  { user_id, risk_level, ... }
  └────┬─────┘
       │
       ▼
  ┌──────────┐
  │ Workflow  │  HealthAlertWorkflow
  │ 工作流    │
  │          │
  │  Step 1: │  把 Command 包装成 MessageEnvelope
  │  Step 2: │  调用 UseCase（执行领域逻辑）
  │  Step 3: │  收集 FeedbackEvent
  │  Step 4: │  触发 Reporting
  │  Step 5: │  返回结果
  └──────────┘
       │
       ▼
  ┌──────────┐
  │ UseCase   │  RunHealthAlertFlowUseCase
  │ 用例      │
  │          │
  │  干什么？ │  1. 消息入聚合器
  │          │  2. 聚合器生成摘要
  │          │  3. 决策核心分发
  │          │  4. 领域服务分诊
  │          │  5. 写入历史
  └──────────┘
```

**CQRS 在这里体现得很清楚：**

```
写路径：                          读路径：

Command                           Query
  │                                 │
  ▼                                 ▼
FlowWorkflow                       QueryWorkflow
  │                                 │
  ▼                                 ▼
RunXxxFlowUseCase                  RunXxxQueryUseCase
  │                                 │
  ▼                                 ▼
写入 SQLite                        从 SQLite 读取
```

**写和读是完全分开的两条路，互不干扰。**

面试时这样说：

> "应用层负责业务流程编排，核心概念是 Command/Query/UseCase/Workflow。写操作走 Command + FlowWorkflow，读操作走 Query + QueryWorkflow，这就是 CQRS。Workflow 把多个 UseCase 串起来，UseCase 调用领域服务和平台组件，但不包含业务规则本身。"

## 知识点 11：用户交互工作流 —— 系统的"前台大门"

前面讲了很多内部机制，现在看**用户是怎么进入系统的**。

整个系统有一个统一的入口，叫 `UserInteractionWorkflow`，所有用户请求都从这进：

```
用户消息
  │
  ▼
┌─────────────────────────────────────────────────────┐
│              UserInteractionWorkflow                  │
│                  （统一前台大门）                       │
│                                                     │
│  ┌─────────┐  ┌──────────┐  ┌─────────┐  ┌──────┐ │
│  │ Step 1  │→│ Step 2   │→│ Step 3  │→│Step 4│ │
│  │ 输入    │  │ 意图识别  │  │ 意图分解 │  │ 路由 │ │
│  │ 预处理  │  │          │  │ (需要时) │  │ 分发 │ │
│  └─────────┘  └──────────┘  └─────────┘  └──────┘ │
│                                                     │
└─────────────────────────────────────────────────────┘
       │
       ▼
  ┌──────────┐ ┌──────────┐ ┌──────────┐ ┌──────────┐
  │ health   │ │daily_life│ │mental_care│ │profile   │
  │ workflow │ │ workflow │ │ workflow │ │memory wf │
  └──────────┘ └──────────┘ └──────────┘ └──────────┘
```

每一步干什么？

**Step 1：输入预处理**

```
原始输入："  我   头晕  "     → 处理后："我 头晕"

做了什么？
  ✓ 去掉首尾空格
  ✓ 连续空白合并成一个
  ✓ 保留原始输入（方便调试）
  ✓ 空消息拦截
  ✓ 特殊字符标记
```

**Step 2：意图识别**

```
"我头晕" → 这是哪种意图？

  可以显式告诉系统：event_type = "health_alert"
  也可以不说，让系统自己判断

  系统怎么判断？→ 下一节重点讲
```

**Step 3：意图分解（仅复杂输入时触发）**

```
"我心里慌，药还要不要天天吃" → 这是复合意图！

  第一层识别到：这同时涉及心理 + 用药
  → 触发第二层分解
  → 拆成：mental_care_checkin + health_alert
```

**Step 4：路由分发**

```
识别出 health_alert
  → 分发到 HealthAlertWorkflow
  → 返回结果

识别出 daily_life_checkin
  → 分发到 DailyLifeCheckInWorkflow
  → 返回结果

无法识别
  → 返回 clarification_needed（澄清响应）

复合意图未分解
  → 返回 decomposition_needed（需要拆分）
```

**统一返回格式：**

```
┌─────────────────────────────────────────┐
│         UserInteractionResponse          │
│                                         │
│  accepted: true/false     ← 系统接受了吗  │
│  routed_event_type: "HEALTH_ALERT"       │
│                           ← 路由到哪了   │
│  response: {...}          ← 领域返回结果  │
│  actions: [...]           ← 系统要做什么  │
│  runtime_signals: [...]   ← 运行时信号   │
│  memory_updates: {...}    ← 记忆更新了啥  │
│  intent: {...}            ← 意图识别详情  │
│  preprocessing: {...}     ← 预处理信息   │
└─────────────────────────────────────────┘
```

不管走哪个领域，返回格式都一样，前端/调用方不用关心内部差异。

面试时这样说：

> "系统有统一的用户交互入口 UserInteractionWorkflow，内部四步流水线：预处理、意图识别、意图分解、路由分发。所有领域返回统一的交互响应格式，调用方不需要关心内部路由差异。"

## 知识点 12：意图识别 —— 系统的"耳朵"

这是你面试时**最容易出彩**的部分，因为涉及 LLM、BERT、规则引擎的组合使用。

用户发来一条消息，系统怎么知道他想干什么？

```
用户说："我头晕"
         │
         ▼
  ┌───────────────────────────────────┐
  │        第一层：意图识别              │
  │                                   │
  │  "这句话属于哪个类别？"              │
  │                                   │
  │  可选方案：                         │
  │  📏 rule_based   规则匹配（默认）    │
  │  🧠 bert         本地 BERT 模型     │
  │  🔀 hybrid       混合策略           │
  └───────────────────────────────────┘
```

### 方案一：规则匹配（rule_based）

```
就是 if-else，但比简单 if-else 聪明一点：

关键词匹配：
  "头晕" "摔倒" "胸痛"      → health_alert
  "买菜" "做饭" "出门"      → daily_life_checkin
  "焦虑" "孤单" "不开心"    → mental_care_checkin
  "记住" "我喜欢" "remember" → profile_memory_update
  "你还记得" "我之前说过"    → profile_memory_query

优点：确定性高，全量 288/288 通过
缺点：覆盖不了没见过的表达
```

### 方案二：BERT 模型（bert）

```
本地 6 类分类模型：

  输入：一句话
  输出：类别 + 置信度

  "我头晕" → health_alert, confidence=0.92 ✓
  "我想喝茶" → profile_memory_update, confidence=0.71 ⚠️

  置信度低于阈值 → fallback 到规则匹配
  模型加载失败   → fallback 到规则匹配
  推理异常      → fallback 到规则匹配
```

**这里有个关键设计：BERT 不是独立运行的，它永远有规则兜底。**

```
"我头晕" 的识别过程（bert 模式）：

  BERT 识别 → health_alert, confidence=0.92
  置信度 > 阈值(0.65) → ✅ 用 BERT 的结果

"我想喝茶" 的识别过程（bert 模式）：

  BERT 识别 → profile_memory_update, confidence=0.45
  置信度 < 阈值(0.65) → ⚠️ 低置信 fallback 到 rule_based
  rule_based 识别 → ✅ 用规则的结果
```

### 还有：hard-case guard

```
BERT 有些场景会"高置信误判"：

  "药吃了以后肚子不舒服"
  BERT 可能判为 daily_life（因为"肚子"像日常）
  但实际应该是 health_alert

  所以加了一层 hard-case guard：
  用规则先拦截已知的易混淆场景
  拦截到了 → 直接纠正
  没拦截到 → 再用 BERT 的结果
```

### 完整的三道防线：

```
用户消息："药吃了以后肚子不舒服"
         │
         ▼
  ┌────────────────┐
  │ 1. 复合意图检测  │  ← 是否包含多种意图信号？
  │  不是复合意图    │     不是 → 继续
  └───────┬────────┘
          ▼
  ┌────────────────┐
  │ 2. hard-case   │  ← 已知易混淆场景拦截
  │    guard       │     "药+不适" → health_alert ✅
  │  命中！直接纠正  │
  └────────────────┘

  如果 hard-case 没命中：
          ▼
  ┌────────────────┐
  │ 3. BERT 推理   │  ← 模型识别
  │  置信度够高？    │     够 → 用 BERT 结果
  │  不够？         │     不够 → fallback 到 rule_based
  └────────────────┘
```

### 三种方案怎么选？

不是根据用户输入动态选择的，而是通过环境变量在启动时配置的。

```
VITALAI_INTENT_RECOGNIZER = rule_based   ← 默认，全用规则
VITALAI_INTENT_RECOGNIZER = bert         ← 优先 BERT，失败 fallback 到规则
VITALAI_INTENT_RECOGNIZER = hybrid       ← 预留，当前和 bert 行为类似
```

也就是说，**方案是运维配的，不是系统选的**：

```
┌─────────────────────────────────────────────────┐
│                                                 │
│  部署时决定用哪个方案，不是运行时动态切换           │
│                                                 │
│  开发/测试环境：rule_based（稳定、快、不需要GPU）   │
│                                                 │
│  有 GPU 的环境：bert（更智能，有兜底）              │
│                                                 │
│  用户输入本身不会决定走哪个方案                     │
│                                                 │
└─────────────────────────────────────────────────┘
```

但是！在 **bert 模式下**，系统会根据每条输入的**识别结果**动态决定最终用谁：

```
VITALAI_INTENT_RECOGNIZER = bert 时：

  每条用户输入进来：

  ┌──────────┐    成功且置信度高    ┌──────────┐
  │          │ ─────────────────→ │ 用 BERT   │
  │  BERT    │                    │ 结果      │
  │  推理    │                    └──────────┘
  │          │    失败或低置信      ┌──────────┐
  │          │ ─────────────────→ │ fallback  │
  └──────────┘                    │ 到规则    │
                                  └──────────┘

  所以 bert 模式其实是 "BERT 优先 + 规则兜底"
  不是纯 BERT
```

### BERT 和 LLM 拆解是什么关系？

这两个**不是竞争关系，是流水线上的两道工序**，解决的问题完全不同：

```
第一层 BERT 回答的问题：
  "这句话属于哪个类别？" → 单选题

第二层 LLM 拆解回答的问题：
  "这句话同时包含好几个意思，怎么拆？" → 解答题
```

举个具体例子：

```
用户说："我头晕"

  第一层 BERT：
    这是一句话，一个意思 → health_alert ✅
    不需要第二层，直接路由

用户说："我心里慌，那个药我到底要不要天天吃啊"

  第一层 BERT：
    这句话有两个意思！
    心里慌 → mental_care
    药要不要吃 → health_alert
    我分不清该去哪个科 😵
    → 返回 decomposition_needed（需要拆分）

  第二层 LLM：
    好的，我帮你拆：
    子任务1：mental_care_checkin（心理关怀）
    子任务2：health_alert（用药咨询）
    风险标记：用药相关，需要谨慎
```

**流程图：**

```
所有用户输入
     │
     ▼
  第一层（BERT 或 规则）
     │
     ├── 能识别出单一意图 ──→ 直接路由到对应领域 ✅
     │
     └── 识别出复合意图 ──→ 返回 decomposition_needed
                              │
                              ▼
                         第二层（LLM 拆解）
                              │
                              └── 拆成多个子任务
                                   （但当前不直接执行，只返回拆分结果）
```

面试时这样说：

> "意图识别采用三道防线：先检测复合意图，再走 hard-case guard 拦截已知易混淆场景，最后才走 BERT 推理。BERT 低置信时自动 fallback 到规则引擎，模型加载失败或推理异常也会 fallback。整个链路保证不会因为模型问题导致识别失败。"

## 知识点 13：第二层意图拆解 —— LLM 怎么拆复合意图

第一层发现复合意图后交给第二层，现在看第二层具体怎么工作。

### LLM 收到什么，产出什么？

```
LLM 收到的输入：
  "我心里慌，那个药我到底要不要天天吃啊"
  + 系统 prompt（告诉 LLM 要按什么格式输出）

LLM 产出的输出（结构化 JSON）：
  ┌─────────────────────────────────────────┐
  │  primary_task:                           │
  │    intent: "health_alert"                │
  │    priority: "high"                      │
  │    confidence: 0.85                      │
  │                                         │
  │  secondary_tasks:                        │
  │    - intent: "mental_care_checkin"       │
  │      priority: "normal"                  │
  │      confidence: 0.7                     │
  │                                         │
  │  risk_flags:                             │
  │    - severity: "high"                    │
  │      category: "medication"              │
  │      description: "涉及用药决策，需谨慎"   │
  └─────────────────────────────────────────┘
```

### 但是！LLM 的输出不能直接用

LLM 有时会胡说八道，所以加了一道**校验器**：

```
LLM 的原始输出
       │
       ▼
┌──────────────────────────────────────┐
│         Schema Validator             │
│                                      │
│  检查什么？                            │
│  ✓ primary_task 不能为空              │
│  ✓ intent 必须是合法值                 │
│    （只能是系统支持的5+1种意图）         │
│  ✓ priority 必须在合法范围             │
│  ✓ confidence 必须在 0~1 之间          │
│  ✓ risk severity 必须是合法值          │
│                                      │
│  不合法 → ❌ 拒绝，只返回校验问题        │
│  合格   → ✅ 通过，进入下一步           │
└──────────────────────────────────────┘
```

```
举个例子：

LLM 输出：
  primary_task: null        ← 没有主任务！
  intent: "shopping"        ← 系统根本不支持这个意图！

校验结果：❌ 拒绝
  - primary_task 缺失
  - intent "shopping" 不合法

这个结果不会进入路由，不会执行任何领域 workflow
```

### 校验通过后，还要过一道"路由守门"

```
合法的拆解结果
       │
       ▼
┌──────────────────────────────────────┐
│      Routing Guard 路由守门            │
│                                      │
│  规则：                               │
│  ✓ 低风险 + 高置信 → 产生"候选"        │
│    （注意：只是候选，不直接执行）         │
│  ✓ 高风险             → 阻断          │
│  ✓ 低置信             → 阻断          │
│  ✓ 缺少 primary_task  → 阻断          │
│  ✓ 需要澄清           → 返回澄清问题    │
└──────────────────────────────────────┘
```

### 完整的四道关卡：

```
"我心里慌，药还要不要天天吃"
         │
         ▼
  ┌─────────────┐
  │ 1. LLM 拆解  │  拆成 mental_care + health_alert
  └──────┬──────┘
         ▼
  ┌─────────────┐
  │ 2. Schema   │  输出格式合法？✅
  │    校验      │
  └──────┬──────┘
         ▼
  ┌─────────────┐
  │ 3. 路由守门  │  用药相关 = 高风险 → ⚠️ 阻断
  │             │  不直接执行，返回候选信息
  └──────┬──────┘
         ▼
  ┌─────────────┐
  │ 4. 返回响应  │  accepted: false
  │             │  error: decomposition_needed
  │             │  告诉调用方拆分结果和风险
  └─────────────┘
```

**为什么拆完了还不执行？因为安全第一。** 养老场景涉及健康和用药，宁可多确认一次，也不可误操作。

### 没通过 ≠ 卡住，而是"告诉调用方，你自己决定"

```
整个流程不是这样：

  用户 → 系统 → ❌ 被挡住了 → 卡在那了 😵

而是这样：

  用户 → 系统 → ❌ 被挡住了 → 把拆分结果和原因都返回给调用方
                                       │
                                       ▼
                              调用方自己决定怎么办
```

**具体返回什么？**

```
用户说："我心里慌，药还要不要天天吃"

系统返回：

  accepted: false                          ← 没有直接执行
  error: decomposition_needed              ← 原因：需要拆分
  
  error_details: {
    decomposition: {                       ← 拆分结果
      status: "pending_second_layer",
      candidate_tasks: [                   ← 拆出来的任务
        { intent: "health_alert", priority: "high" },
        { intent: "mental_care_checkin", priority: "normal" }
      ],
      risk_flags: [                        ← 风险标记
        { severity: "high", category: "medication" }
      ]
    },
    decomposition_guard: {                 ← 守门结果
      status: "blocked",
      candidate_ready: false,
      blocked_reasons: ["high_risk_medication"],  ← 为什么挡
      clarification_question: "您提到的用药问题需要医生确认，是否继续？"
    }
  }
```

**调用方拿到这个结果后，有三种选择：**

```
选择1：弹出确认框
  "系统检测到您提到了用药问题，
   需要医生确认后才能继续，
   是否联系医生？"
  用户点"是" → 调用方再发一次请求，带上确认标记

选择2：转人工
  "这个问题比较复杂，已转给人工客服处理"

选择3：先处理低风险的部分
  心理关怀部分是低风险的 → 先执行
  用药部分高风险 → 等确认
```

**总结一句话：被挡住不是卡住，是把拆分结果、风险标记、澄清问题都交还给调用方，让调用方决定是确认后继续、转人工、还是先处理低风险部分。**

面试时这样说：

> "第二层意图拆解用 LLM 把复合输入拆成多个子任务，但 LLM 输出必须过两道关卡：第一道是 Schema 校验，拒绝非法意图和越界值；第二道是路由守门，高风险和低置信的拆解结果只产生候选不直接执行。当前第二层是非执行态，拆分结果供人工确认。"

## 知识点 14：接口层 —— 三种入口，同一套内核

前面讲了系统内部怎么运转，现在看**外面怎么进来**。

接口层有三种入口通道，但它们**内部调用的是同一套应用层代码**：

```
                    ┌─────────────────────────┐
                    │                         │
    🌐 HTTP请求 ──→ │                         │
                   │    应用层（同一套内核）     │
    ⏰ 定时任务 ──→ │                         │
                   │                         │
    📨 消息消费 ──→ │                         │
                    │                         │
                    └─────────────────────────┘

三种入口，三种角色，但干的活都一样
```

### 为什么需要三种入口？

```
🌐 HTTP 请求（API 角色）
  老人/家属/APP 主动发来的请求
  → 实时响应
  → 比如："我头晕"

⏰ 定时任务（Scheduler 角色）
  系统自己到时间就触发的任务
  → 不需要人操作
  → 比如：每天早上8点检查所有老人昨晚有没有异常

📨 消息消费（Consumer 角色）
  其他系统发来的事件通知
  → 被动接收
  → 比如：医院系统发来"老人刚刚做完手术"
```

### 三种角色的差异

```
┌────────────┬──────────────┬──────────────┬──────────────┐
│            │  API 角色     │ Scheduler角色 │ Consumer角色  │
├────────────┼──────────────┼──────────────┼──────────────┤
│ 谁触发的    │ 用户主动请求   │ 定时器自动触发 │ 外部系统推送   │
├────────────┼──────────────┼──────────────┼──────────────┤
│ 需要回执？  │ 可选          │ 不需要        │ 需要          │
│ require_ack│              │ false        │ true         │
├────────────┼──────────────┼──────────────┼──────────────┤
│ 消息存活    │ 中等          │ 较长          │ 较短          │
│ TTL        │ 120秒        │ 300秒        │ 60秒         │
├────────────┼──────────────┼──────────────┼──────────────┤
│ 报告生成    │ ✅ 开启       │ ❌ 关闭       │ ❌ 关闭       │
├────────────┼──────────────┼──────────────┼──────────────┤
│ 运行时信号  │ ✅ 开启       │ ❌ 关闭       │ ❌ 关闭       │
├────────────┼──────────────┼──────────────┼──────────────┤
│ 适合做什么  │ 实时交互      │ 批量巡检      │ 事件驱动      │
└────────────┴──────────────┴──────────────┴──────────────┘
```

### 关键设计：接口层必须极薄

```
❌ 胖接口：
  接口层里写了：参数校验 + 业务逻辑 + 数据库操作 + 决策
  → 换个入口方式（比如从HTTP换成消息），逻辑全废了

✅ 薄接口：
  接口层只做三件事：
  1. 接收输入
  2. 调用应用层
  3. 返回结果

  业务逻辑全在应用层
  → 不管从哪进来，都走同一套逻辑
  → 换入口方式零成本
```

```
┌──────────────────────────────────────────────────┐
│                                                  │
│  HTTP 接口：                                      │
│    接收 JSON → 调用 assembly.build_workflow()     │
│             → 返回 JSON                          │
│                                                  │
│  Scheduler 任务：                                 │
│    到时间了 → 调用 assembly.build_workflow()      │
│            → 记日志                              │
│                                                  │
│  Consumer 消费：                                  │
│    收到消息 → 调用 assembly.build_workflow()      │
│            → 发 ACK                              │
│                                                  │
│  三种入口都只做"接收→调用→返回"，业务全在应用层    │
│                                                  │
└──────────────────────────────────────────────────┘
```

面试时这样说：

> "接口层支持三种入口通道：HTTP API、定时任务和消息消费，对应三种运行角色。关键设计是接口层极薄，只做接收输入、调用应用层、返回结果，业务逻辑全在应用层。这样不管从哪种方式接入，都复用同一套核心代码，换入口方式零成本。"

## 知识点 15：Assembly 装配器 —— 怎么把所有零件拼起来

前面讲了领域层、平台层、应用层、接口层，但有一个问题：

```
领域服务需要仓储
仓储需要数据库连接
工作流需要领域服务 + 平台组件
接口层需要工作流

谁负责把这些东西组装在一起？
  → Assembly 装配器
```

### 打个比方：拼乐高

```
你有一堆乐高零件：

  🏥 健康领域模块
  🍚 日常生活模块
  💚 心理关怀模块
  🧠 画像记忆模块
  📨 消息信封
  ⚖️ 仲裁器
  🚨 中断器
  💾 数据库连接
  🤖 BERT 模型
  ...

Assembly 就是那个"拼乐高的人"：

  把数据库连接塞进仓储
  把仓储塞进领域服务
  把领域服务塞进工作流
  把工作流交给接口层

  拼完之后，接口层说"我要健康工作流"
  Assembly 就递给它一个拼好的、可以直接用的
```

### Assembly 的核心设计

```
┌─────────────────────────────────────────────────┐
│            ApplicationAssembly                   │
│                                                 │
│  持有的零件（惰性初始化，用到才创建）：             │
│                                                 │
│  💾 health_repo                                 │
│  💾 daily_life_repo                             │
│  💾 mental_care_repo                            │
│  💾 profile_memory_repo                         │
│  🤖 intent_recognizer                           │
│  🧠 intent_decomposer                           │
│  📊 reporting_service                           │
│  ...                                            │
│                                                 │
│  提供的拼装方法：                                  │
│                                                 │
│  build_health_alert_workflow()      → 拼好的     │
│  build_daily_life_workflow()        → 拼好的     │
│  build_mental_care_workflow()       → 拼好的     │
│  build_profile_memory_workflow()    → 拼好的     │
│  build_user_interaction_workflow()  → 拼好的     │
│  build_health_alert_query_workflow() → 拼好的    │
│  ...                                            │
└─────────────────────────────────────────────────┘
```

### 怎么从环境变量到拼好的系统？

```
启动时一条链：

  环境变量 (.env)
       │
       ▼
  ApplicationAssemblyEnvironment.from_environment()
    → 读取所有配置（用什么意图识别器、数据库路径、LLM配置...）
       │
       ▼
  .to_config()
    → 生成工厂方法注册表（每个零件怎么创建）
       │
       ▼
  ApplicationAssembly(config)
    → 拿到装配器实例
       │
       ▼
  assembly.build_xxx_workflow()
    → 按需拼装，用到才创建
```

### 惰性初始化 —— 用到才创建，不浪费

```
不是启动时把所有零件都创建好，而是：

  调用 build_health_workflow() 时
    → 才创建 health_repo
    → 才创建 health_service
    → 才拼装 workflow

  如果只用了 health，不会创建 daily_life 的任何东西

  像点菜：点了才做，不是把菜单上的全做一遍
```

### 不同角色，不同拼法

```
API 角色：     所有零件都拼上（需要响应所有请求）
Scheduler 角色： 关掉 reporting 和 runtime_signals（不需要报告和信号）
Consumer 角色：  关掉 reporting 和 runtime_signals（不需要报告和信号）

同一套零件，不同角色少拼几块
```

### 会不会运行很慢？

不会，因为"惰性"只影响第一次，之后都是复用。

```
第一次调用 build_health_workflow()：
  创建 health_repo     ← 花时间
  创建 health_service  ← 花时间
  拼装 workflow        ← 花时间
  → 稍微慢一点（毫秒级）

第二次、第三次、第N次调用：
  直接返回上次创建好的 workflow
  → 速度一样快，没有区别
```

真正的性能瓶颈：

```
┌──────────────────────────────────────────────┐
│                性能瓶颈排名                     │
│                                              │
│  🐢 LLM 调用（意图拆解时）       1~5秒        │
│  🐢 BERT 推理（意图识别时）      0.1~0.5秒    │
│  🐌 SQLite 读写                  0.01~0.05秒 │
│  ⚡ Assembly 对象创建（仅首次）   <0.01秒      │
│  ⚡ 消息信封封装                  <0.001秒     │
│  ⚡ 决策核心分发                  <0.001秒     │
│                                              │
│  Assembly 创建的开销几乎可以忽略               │
└──────────────────────────────────────────────┘
```

面试时这样说：

> "Assembly 是系统的依赖注入容器，负责把仓储、领域服务、平台组件和工作流拼装在一起。采用惰性初始化，用到才创建，避免启动时全量构建。不同运行角色（API/Scheduler/Consumer）共享同一套零件，但按角色配置关闭不需要的功能。"

## 知识点 16：一条请求的完整旅程 —— 把所有知识点串起来

前面讲了很多零件，现在看一条请求从头到尾怎么走完。我用两个场景来走：一个简单的，一个复杂的。

### 场景1：简单请求 —— "我头晕"

```
👴 老人说："我头晕"
│
▼
① 接口层：HTTP POST /vitalai/interactions
   收到：{ user_id: "elder-001", message: "我头晕" }
   做：构建 UserInteractionCommand → 交给应用层
│
▼
② 应用层：UserInteractionWorkflow
│
   ②-1 输入预处理
       "我头晕" → "我头晕"（没有多余空格，不需要改）
│
   ②-2 意图识别（第一层）
       规则/BERT → health_alert, confidence=0.92
       单一意图，不需要拆分 ✅
│
   ②-3 路由分发
       识别出 health_alert → 分发到 HealthAlertWorkflow
│
▼
③ HealthAlertWorkflow 执行
│
   ③-1 构建 MessageEnvelope
       priority=CRITICAL, from="user", payload={risk_level: "high"}
│
   ③-2 RunHealthAlertFlowUseCase 执行
   │
       ③-2-1 消息入聚合器 EventAggregator.ingest()
       ③-2-2 聚合器生成摘要 EventAggregator.summarize()
              └→ 同时推入信号桥接 → 可观测记录 + 安全审查
       ③-2-3 决策核心 DecisionCore.process_summary()
              → 查表：health_alert → 调用健康分诊服务
       ③-2-4 领域服务 HealthAlertTriageService.triage()
              → 输出：决策消息 + 反馈事件 + 仲裁意图
       ③-2-5 写入历史 HealthAlertRepository.save()
              → 存到 .runtime/health.sqlite3
│
   ③-3 快照（CRITICAL 级别自动触发）
       SnapshotStore.save() → 保存当前系统状态
│
   ③-4 收集反馈 + 报告
│
▼
④ 返回响应

{
  accepted: true,
  routed_event_type: "HEALTH_ALERT",
  response: { risk_level: "high", ... },
  intent: { primary_intent: "health_alert", source: "rule_based" },
  runtime_signals: [...],
  memory_updates: null
}
```

### 场景2：复杂请求 —— "我心里慌，药还要不要天天吃"

```
👴 老人说："我心里慌，药还要不要天天吃"
│
▼
① 接口层 → ② 应用层：UserInteractionWorkflow
│
   ②-1 预处理："我心里慌，药还要不要天天吃" → 无变化
│
   ②-2 意图识别（第一层）
       检测到复合意图！（心理+用药）
       → 返回 decomposition_needed ❌
│
   ②-3 意图拆解（第二层）
   │
       LLM 拆解：
         primary_task: health_alert (用药)
         secondary_task: mental_care_checkin (心理)
         risk_flags: 高风险(涉及用药)
   │
       Schema 校验：✅ 格式合法
   │
       路由守门：❌ 高风险，阻断
         → 只返回候选，不直接执行
│
▼
③ 返回响应

{
  accepted: false,
  error: "decomposition_needed",
  intent: { requires_decomposition: true },
  error_details: {
    decomposition: {
      candidate_tasks: [health_alert, mental_care_checkin],
      risk_flags: [{ severity: "high", category: "medication" }]
    },
    decomposition_guard: {
      status: "blocked",
      blocked_reasons: ["high_risk_medication"],
      clarification_question: "用药问题需医生确认，是否继续？"
    }
  }
}
```

### 两张图对比

```
简单请求的旅程：

  用户 → 接口层 → 预处理 → 意图识别(单一) → 路由
    → 工作流 → 聚合器 → 决策核心 → 领域服务
    → 写历史 → 拍快照 → 返回 ✅

复杂请求的旅程：

  用户 → 接口层 → 预处理 → 意图识别(复合!)
    → LLM拆解 → Schema校验 → 路由守门(高风险!)
    → 返回拆分结果+澄清问题 ⚠️

  关键区别：复杂请求不会自动执行，而是交给人决定
```

面试时这样说：

> "简单请求走完整的预处理→识别→路由→领域处理→历史写入→快照→返回链路。复杂请求在意图识别阶段就被拦住，走LLM拆解→校验→守门链路，高风险只返回候选不直接执行，保证养老场景的安全性。"

## 知识点 17：反馈层 —— 做完事怎么"学经验"

系统每次处理完一件事，不是干完就完了，还会**给自己留反馈**，就像人做完事后复盘一样。

### 三层反馈机制

```
┌──────────────────────────────────────────────────┐
│                                                  │
│  L1 即时反馈   做完一件事，立刻记录                 │
│  "刚才发生了什么"                                  │
│                                                  │
│  L2 聚合反馈   积累一段时间，总结规律               │
│  "最近一周老人经常头晕"                             │
│                                                  │
│  L3 学习反馈   根据规律调整策略                     │
│  "既然经常头晕，以后自动加强健康监测"                 │
│                                                  │
└──────────────────────────────────────────────────┘
```

### 用生活比喻：

```
L1 即时反馈 = 值班日记
  "4月16日 10:05，老人报告头晕，已处理"

L2 聚合反馈 = 周报
  "本周老人报告头晕5次，比上周增加3次，
   建议关注血压变化"

L3 学习反馈 = 流程改进
  "老人经常头晕，以后每天主动询问身体状况，
   不用等老人自己说"
```

### 代码里的 FeedbackEvent 长什么样？

```
┌─────────────────────────────────────────┐
│         FeedbackEvent                    │
│                                         │
│  source_agent: "health_triage"  ← 谁反馈 │
│  feedback_layer: L1             ← 第几层  │
│  feedback_type: "alert_processed" ← 类型 │
│  confidence: 0.9               ← 置信度  │
│  payload: {                    ← 具体内容 │
│    risk_level: "high",                   │
│    action_taken: "escalated"             │
│  }                                       │
│  trace_id: "trace-xxx"          ← 追踪ID │
└─────────────────────────────────────────┘
```

### 反馈怎么被用起来？

```
领域服务处理完 → 产生 FeedbackEvent
       │
       ▼
Workflow 收集反馈
       │
       ├── → 推入 Reporting（生成报告）
       ├── → 推入 Observability（记录日志）
       └── → 推入聚合反馈（为 L2/L3 积累素材）
```

面试时这样说：

> "系统有三层反馈机制：L1 即时反馈记录每次处理结果，L2 聚合反馈总结一段时间内的规律，L3 学习反馈根据规律调整策略。每次领域处理完都会产出 FeedbackEvent，推入报告、可观测和聚合链路。"

---

## 知识点 18：可观测 + 安全 —— 两个横切面

这两个不是某个业务流程的一部分，而是**横跨所有流程的"监控探头"**，每个关键动作都会同时经过它们。

### 可观测层 —— 系统的"黑匣子"

```
飞机有黑匣子，记录飞行中的一切
VitalAI 也有自己的"黑匣子"

每当系统发生关键动作，自动记录：

  ┌──────────────────────────────────────────┐
  │        ObservabilityCollector             │
  │                                          │
  │  记录什么？                                │
  │                                          │
  │  📥 消息接收   谁发了消息，什么时候收到的     │
  │  📤 决策输出   做了什么决策，结果是什么       │
  │  📸 快照保存   什么时候拍了快照              │
  │  🚨 中断触发   什么中断了，优先级多少         │
  │  🔄 反馈产生   什么反馈，置信度多少          │
  │  ⚖️ 仲裁裁决   谁和谁冲突，结果是什么        │
  │  🔀 故障转移   什么时候切换，从谁切到谁       │
  └──────────────────────────────────────────┘
```

```
为什么重要？

  出问题时：查黑匣子，还原当时发生了什么
  优化时：看黑匣子，发现瓶颈在哪里
  审计时：看黑匣子，证明系统做了该做的事
```

### 安全层 —— 系统的"安检员"

```
每当系统处理用户数据，自动扫描：

  ┌──────────────────────────────────────────┐
  │        SensitiveDataGuard                 │
  │                                          │
  │  扫描什么？                                │
  │                                          │
  │  🔍 消息载荷   payload 里有没有敏感信息？    │
  │  🔍 中断信号   中断原因里有没有隐私？        │
  │  🔍 快照内容   快照里有没有不该存的？        │
  │                                          │
  │  怎么处理？                                │
  │  🏷️ 标记   标记哪些字段是敏感的             │
  │  🔒 脱敏   输出时自动遮挡                  │
  │  🚫 拦截   超标的数据不让通过               │
  └──────────────────────────────────────────┘
```

### 关键设计：信号桥接自动分发，不需要每个模块自己调

```
❌ 没有信号桥接：
  每个领域服务自己写：
    observability.record(...)
    security.scan(...)
  → 忘了写就漏了 😵

✅ 有信号桥接（SignalBridge）：
  每个关键动作自动推入桥接
  桥接同时分发给可观测 + 安全
  
  领域服务不需要知道可观测和安全的存在
  → 不会漏 😎
```

```
所有关键动作
     │
     ▼
┌──────────────┐
│ SignalBridge  │
│  信号桥接     │
└──────┬───────┘
       │
  ┌────┴────┐
  ▼         ▼
👁️ 可观测   🔒 安全
记录日志    扫描脱敏
```

面试时这样说：

> "可观测层和安全层是两个横切面，通过信号桥接自动接入所有关键流程。每个关键动作自动推入桥接，桥接同时分发给可观测记录和安全扫描，领域代码不需要感知这两个横切面的存在，不会遗漏。"

## 知识点 19：Admin 控制面 —— 运维的"后台遥控器"

系统正常运行时，用户通过 `/vitalai/interactions` 进入。但运维人员需要一些**特殊的控制接口**，能直接观察系统内部状态、甚至模拟故障。

### 控制面接口

```
普通用户走的路：
  POST /vitalai/interactions        ← 正常业务

运维走的路：
  POST /vitalai/admin/runtime-diagnostics/api           ← 查看运行时诊断
  POST /vitalai/admin/runtime-diagnostics/api/health-failover  ← 模拟故障转移
```

### 诊断接口能看什么？

```
调用 diagnostics 后返回：

┌──────────────────────────────────────────┐
│          RuntimeDiagnostics               │
│                                          │
│  📊 策略快照                              │
│     当前入站策略（优先级、TTL、ACK设置）    │
│                                          │
│  📸 最新快照                              │
│     系统最近的状态存档                     │
│                                          │
│  🔄 运行时信号                            │
│     最近发生了哪些关键事件                  │
│                                          │
│  🏥 健康状态                              │
│     各组件是否正常                         │
└──────────────────────────────────────────┘
```

### 故障转移演练能干什么？

```
模拟主决策核心挂了：
  → 触发中断信号
  → 影子核心从快照恢复
  → 接管成为新的 active
  → 验证故障转移链路是否正常

就像消防演练：
  不是真着火了
  而是定期演练，确保真着火时能正常响应
```

### 安全设计：不是谁都能调

```
┌─────────────────────────────────────────────┐
│           Admin 接口的安全防护                 │
│                                             │
│  1️⃣ 默认关闭                                │
│     VITALAI_RUNTIME_CONTROL_ENABLED = true   │
│     不配这个，接口直接 403                     │
│                                             │
│  2️⃣ 必须带 Token                            │
│     请求头：X-VitalAI-Admin-Token: xxx       │
│     不带或错误 → 拒绝访问                     │
│                                             │
│  3️⃣ Token 从环境变量读取                     │
│     VITALAI_ADMIN_TOKEN = "dev-admin-token"  │
│     不硬编码在代码里                          │
│                                             │
│  三道门：                                    │
│  没开控制面 → 🚫                             │
│  没带Token → 🚫                             │
│  Token错误 → 🚫                             │
│  全通过 → ✅                                 │
└─────────────────────────────────────────────┘
```

面试时这样说：

> "系统有独立的 Admin 控制面，提供运行时诊断和故障转移演练。安全上三道门：默认关闭控制面、必须携带 Admin Token、Token 从环境变量读取不硬编码。这保证了运维能力可用但不会被滥用。"
