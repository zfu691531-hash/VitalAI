## 一、项目概况

- **项目名称**：校园AI教务助手平台（aistu）
- **定位**：面向中小学教育场景的一站式教务管理与AI智能辅助一体化平台
- **核心特色**：深度整合学生信息管理系统与8大AI教务智能工具，严格区分学生/教师/校务管理员三级角色权限

---

## 二、技术栈决策

| 层级 | 选型 | 决策理由 |
|---|---|---|
| 后端框架 | FastAPI | 异步高性能，自动Swagger文档，Pydantic原生集成 |
| 数据库 | MySQL 8.0 | 稳定可靠，校园场景数据量适中 |

| UI组件库 | Element Plus | 管理系统组件丰富 |

| CSS预处理 | SCSS | 变量管理、嵌套语法 |

## 三、架构设计演进

### V1架构问题（已修正）

经过二次审查，V1架构存在以下6个缺陷，已全部修正：

1. **AI服务过于臃肿**：V1将8个AI工具写在一个service文件中 → 修正为 `services/ai/` 子包，每个工具独立文件，继承 `base.py` 基类
2. **缺少校规表**：需求中校规问答机器人需要校规数据 → 新增 `school_rule` 表和对应CRUD模块
3. **权限服务与依赖注入重叠**：V1有独立的 `permission_service.py` 和 `deps.py` 中的权限逻辑 → 合并为 `api/deps.py` 中的 `require_role()` 依赖注入
4. **缺少数据库迁移方案**：无Alembic → 添加 `alembic/` 目录和迁移工作流
5. **缺少环境变量管理**：配置硬编码 → 添加 `.env` + pydantic-settings
6. **缺少403/404错误页和文件上传组件**：前端骨架不完整 → 补充 `views/error/` 和 `FileUpload.vue`

### V2最终架构（已执行）

- 后端三层架构：`api/` → `services/` → `database/`
- 前后端对称设计：api/ 模块与前端 api/ 层1:1对应
- AI工具按4种交互模式分类：单次生成(5个)、多轮对话(2个)、DB联动(2个)、算法型(1个)

## 四、数据库设计决策

### 7张表，无外键约束

- user / student / teacher / class / score / ai_history / school_rule
- **不使用外键**：由业务层保证数据一致性，删除更灵活，批量导入不受阻塞
- teacher.class_ids 用逗号分隔字符串存储（设计妥协，班级数<20时够用）
- class.current_count 由业务层在增删改学生时同步维护

### 密码存储方案

需求文档提到MD5，但检查清单明确禁止MD5 → 采用 bcrypt（passlib库），此决策已记录并通知所有工程师。

## 五、AI工具架构

### 基类设计 (`services/ai/base.py`)

- 统一封装大模型API调用（httpx异步客户端）
- 内置超时重试（最多3次，指数退避）+ 降级处理
- 全局单例 `ai_client`

### 8个工具文件

| 文件 | 交互模式 | DB联动 | 备注 |
|---|---|---|---|
| comment_generator | 单次生成 | 可选student_id | 支持批量班级生成 |
| discipline_coach | 单次生成 | 无 | 三段式话术 |
| notice_polisher | 单次生成 | 可选class_id | 标准通知格式 |
| rule_bot | 多轮对话 | school_rule表 | 基于校规内容回答 |
| score_diagnosis | 单次生成 | score表 | 自动读取历史成绩 |
| meeting_planner | 单次生成 | 无 | 完整教案 |
| mock_interview | 多轮对话 | 可选student_id | AI提问→用户回答→点评 |
| group_helper | 单次生成 | student/class表 | 教师/校务双模式 |

## 六、权限体系

三级角色：student(学生) / teacher(教师) / admin(校务管理员)

- **学生**：仅个人数据，校规问答+模拟面试
- **教师**：本班数据，全部AI工具，班级内分组
- **管理员**：全校数据，全部CRUD，校务分班，校规管理

前端双重权限控制：路由守卫(页面级) + v-permission指令(按钮级)

## 八、关键风险与注意事项

1. **AI API调用延迟**：大模型响应可能5-30秒，前端必须loading提示+超时处理
2. **批量导入数据一致性**：逐行校验关联ID有效性，错误明细返回给用户
3. **class_ids逗号分隔方案局限**：班级绑定数>20时需改为关联表
4. **current_count同步风险**：并发操作可能导致计数不一致，需考虑事务+行锁
5. **AI幻觉问题**：检查清单要求对AI生成内容添加"仅供参考"提示
6. **需求文档与检查清单的冲突**：需求提到MD5但检查清单禁止，遵循安全原则选bcrypt
