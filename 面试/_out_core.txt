三、紧急中断机制

当前系统存在一个"紧急中断"模块，但缺乏内部分级和执行规范。就像一个公司只设置了一个紧急联系电话，不论是大楼着火还是打印机卡纸，都打同一个号码、走同一个流程。

用户早上起床，日常行为Agent给用户推送了"今天要完成3个工作任务"的计划。

📌 实例：场景二：用户拒绝建议，系统无感
健康Agent根据用户的心率数据，连续3天都在早上8点推送"建议你今天减少咖啡摄入"。
用户每次都直接划掉，没有采纳。但健康Agent没有收到任何反馈，不知道用户在拒绝。
第4天、第5天，健康Agent继续推送同样的建议。
用户开始觉得这个系统"很烦"，体验变差。
根本原因：用户的行为（拒绝划掉）没有被转化为系统可理解的信号，个人信息库里这条偏好从未被更新。

📌 实例：场景三：执行质量无法评估
反思Agent在每天晚上总结用户的一天。它生成了一段总结文本，Supervisor收到后认为任务完成。
但这段总结的质量其实参差不齐：有时候很精准，有时候完全是在凑字数。
因为没有质量评分机制，Supervisor无法区分"高质量完成"和"低质量完成"。
长期下来，系统无法学习哪些Agent表现好、哪些需要优化。

// ── 用户拒绝行为处理逻辑（伪代码）

function handleUserReject(agentId, suggestionId, userId) {
  // 1. 构造USER_REJECT反馈事件
  const event = {
    event_type: "USER_REJECT",
    agent_id: agentId,           // 例如 "agent_health"
    suggestion_id: suggestionId, // 例如 "reduce_coffee_morning"
    user_id: userId,
    timestamp: now()
  }

  // 2. 发送到消息总线
  eventBus.publish("feedback.user", event)

  // 3. 反馈聚合器处理：查询历史拒绝次数
  const rejectCount = db.query(
    "SELECT count FROM reject_log WHERE agent_id=? AND suggestion_id=?",
    [agentId, suggestionId]
  )

  // 4. 根据拒绝次数更新权重
  if (rejectCount >= 3) {
    // 连续拒绝3次 → 大幅降低该建议权重，未来30天不再推送
    personalInfoDB.updatePreference({
      key: "health.coffee_reduction_suggestion",
      weight: -0.8,
      suppress_days: 30
    })
  } else {
    // 单次拒绝 → 轻微降低权重，调整推送时间
    personalInfoDB.updatePreference({
      key: "health.coffee_reduction_suggestion",
      weight: -0.2,
      adjust_time: "shift_to_afternoon"
    })
  }
}

📌 实例：场景一：紧急等级混乱
情况A：用户被检测到心率突然飙升到160bpm（可能是健康危机）
情况B：日常Agent和健康Agent同时想在早上9点推送消息（任务调度冲突）
当前系统：两种情况都触发"紧急中断"，走一样的处理流程，优先级无区分。
后果：真正的健康危机可能要等任务冲突处理完才被响应；或者任务冲突被当成生命危机一样处理，大量资源浪费。

--- TABLE 15 ---
📌 实例：场景二：中断后系统"失忆"
用户正在专注工作，任务Agent正在执行第3步（共5步）的一个复杂任务。
此时健康Agent检测到用户久坐超过2小时，触发紧急中断，打断任务Agent。
任务Agent被迫停止，它没有保存自己执行到第3步的状态。
中断处理完（用户站起来活动）后，没有恢复机制，任务Agent只能从头开始。
结果：用户需要重新等待任务从第1步重新执行，体验极差。

// ── 中断状态机定义
const InterruptStates = {
  NORMAL:              "NORMAL",            // 系统正常运行
  INTERRUPT_DETECTED:  "INTERRUPT_DETECTED", // 检测到触发信号
  EVALUATING:          "EVALUATING",         // 评估优先级和影响域
  CONFIRMING:          "CONFIRMING",         // 等待用户确认（P1/P2）或计时（P0的3秒窗口）
  EXECUTING:           "EXECUTING",          // 中断逻辑执行中
  RECOVERING:          "RECOVERING",         // 中断结束，各Agent恢复状态
}

// ── 合法状态转换
const transitions = {
  NORMAL:             → INTERRUPT_DETECTED  （收到触发信号）
  INTERRUPT_DETECTED: → EVALUATING          （信号合法）
  EVALUATING:         → CONFIRMING          （评估完成）
                      → NORMAL              （判定为误触发）
  CONFIRMING:         → EXECUTING           （用户确认 或 P0倒计时结束）
                      → NORMAL              （用户3秒内撤销 或 P0误触发撤销）
  EXECUTING:          → RECOVERING          （中断处理完毕）
                      → ESCALATING          （情况恶化，需人工接管）
  RECOVERING:         → NORMAL              （所有Agent恢复完毕）
}

// 场景：健康Agent、任务Agent、习惯Agent同时要在9:00推送
function resolveResourceConflict(intents) {
  // 1. 按goal_weight排序
  const sorted = intents.sort((a, b) => b.goal_weight - a.goal_weight)
  // 结果：健康(0.8) > 任务(0.7) > 习惯(0.5)

  // 2. 检查是否可以合并
  const canMerge = checkMergeCompatibility(sorted)
  if (canMerge) {
    // 合并成一条推送："早安！今日摘要：建议降低工作强度 + 5个待办 + 写日记"
    return mergeNotifications(sorted)
  }

  // 3. 不可合并 → 时间错位：第一个立即推送，后面的延迟
  sorted[0].send_time = "09:00"    // 健康Agent：立即推送
  sorted[1].send_time = "09:15"    // 任务Agent：延迟15分钟
  sorted[2].send_time = "09:30"    // 习惯Agent：延迟30分钟
  return sorted
}

// 目标冲突仲裁示例：任务Agent(0.70) vs 健康Agent(0.85)
function resolveGoalConflict(intentA, intentB) {
  const weightDiff = Math.abs(intentA.goal_weight - intentB.goal_weight)

  if (weightDiff > 0.3) {
    // 权重差值大 → 系统自动仲裁，高权重方获胜
    // 本例：健康(0.85) - 任务(0.70) = 0.15，差值≤0.3，进入else
    return autoDecide(intentA, intentB)
  } else {
    // 权重差值小（≤0.3）→ 上报用户，让用户裁决
    // 以通俗语言描述冲突，例如：
    notifyUser({
      message: "健康助手建议你休息一下，但你的任务还差一步就完成了。你想怎么做？",
      options: [
        { label: "先休息20分钟再完成任务", action: "health_first" },
        { label: "完成这步任务再去休息", action: "task_first" },
        { label: "让系统帮我决定",       action: "auto" }
      ],
      record_as_preference: true  // 用户选择将记录为偏好，下次自动应用
    })
  }
}

📌 实例：早上7:00 用户醒来，系统的完整工作流程
═══ 通信层（第一层）启动 ═══
① 07:00:00 系统醒来，健康Agent读取昨夜睡眠数据，通过同步Request-Reply查询个人信息库
② 07:00:01 健康Agent发现用户昨晚睡了5.5小时（低于7小时阈值），构造EVENT消息，发布到 topic.health.data.updates
③ 日常任务Agent、精神Agent、Supervisor 均已订阅该Topic，同时收到健康数据更新

═══ 冲突层（第二层）仲裁 ═══
④ 07:00:02 任务Agent意图：推送"今日待办5项，建议现在开始处理"（目标权重0.70）
         健康Agent意图：推送"睡眠不足，建议减少今日任务量"（目标权重0.85）
         习惯Agent意图：推送"晨间日记提醒"（目标权重0.60）
⑤ 冲突检测器：发现三方同时要推送，资源冲突+目标冲突双重触发
⑥ 仲裁结果：健康(0.85)权重最高，且和任务(0.70)差值=0.15≤0.3，上报用户
         通知用户："今天睡眠不足，助手建议减少任务，但你有5个待办要处理。你想怎么安排？"
⑦ 用户选择："先处理3个最重要的，其余推到明天"
         该选择写入个人信息库：睡眠不足时的任务优先级策略=保留TOP3

═══ 反馈层（第三层）运转 ═══
⑧ 任务Agent按用户决策执行3个任务，每完成一个，发送FEEDBACK_POS事件（L1层）
⑨ Supervisor实时消费L1反馈，更新系统状态：今日任务 3/3 → 进行中
⑩ 下午18:00，反馈聚合器触发L1→L2处理：今日完成率100%，睡眠不足天数=1
⑪ 周末，L2→L3模式提取：发现用户睡眠不足时选择"保留TOP3任务"已有4次，
   自动写入L3长期偏好：睡眠不足场景下，自动将任务计划调整为TOP3模式（无需再询问）

═══ 中断层（第四层）监控 ═══
⑫ 全程：中断控制器持续后台监控，今早无异常
   若此时检测到心率突然>150bpm：触发P1中断，Supervisor为所有Agent保存快照，
   启动健康监控强化模式，之前执行到一半的任务状态被完整保存，健康警报处理完后无缝恢复
