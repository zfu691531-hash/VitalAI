一、什么是冲突仲裁层
冲突仲裁层是系统的"协调员"和"裁判"。它坐在通信层（第一层）和反馈层（第二层）之上，在任何Agent行动之前，先检查：这个行动会不会和其他Agent产生矛盾？如果有矛盾，谁的主张更重要？能不能找到双方都能接受的方案？




二、四象限冲突分类体系
冲突仲裁层首先把冲突分为四类，不同类型的冲突有不同的解决策略。就像医院分诊——不同的病症走不同的诊室。



三、意图声明池——行动前先"报备"
冲突仲裁层的第一道防线是"意图声明池"。任何Agent在执行一个会影响王奶奶的行动之前，必须先向冲突检测器提交一份"意图声明"，声明自己想做什么、需要什么资源、什么时候做。

就像王奶奶家里住着的几个帮手，每天早上开一个5分钟的小会，各自说今天的计划——这样大家都知道彼此要做什么，提前协调好，不会发生撞车。




四、目标权重仲裁——谁的主张更重要
当资源冲突无法通过合并或延迟解决，或者目标冲突导致建议相互矛盾时，冲突仲裁层使用"目标权重仲裁"来决定谁说了算。

五、乐观锁——保护共享数据不被乱改
当多个Agent需要同时修改个人信息库里的同一个字段时，如果不加控制，后写入的数据会覆盖先写入的数据，造成信息丢失。乐观锁用"版本号"机制解决这个问题，代价小、效率高。

5.1 为什么叫"乐观"锁
乐观锁的哲学是：先假设"大概率不会发生冲突"，所以不加锁直接操作，只在写入时检查一下"我读到的版本号还对吗"。如果对，写入成功；如果不对，说明有人抢先改了，这时候再处理冲突。
这就像两个家庭成员都想更新同一张纸上的购物清单，约定：写之前先看一下清单是第几版，写完注明变成第几版。如果发现清单版本被别人改了，就先看看别人改了什么，再合并进去。

// 目标权重仲裁逻辑
// 场景：健康Agent(0.85) vs 日常Agent(0.70) 的目标冲突

function arbitrateGoalConflict(intentA, intentB, userId) {
  const weightA = intentA.goal_weight  // 健康Agent = 0.85
  const weightB = intentB.goal_weight  // 日常Agent = 0.70
  const diff    = Math.abs(weightA - weightB)  // = 0.15

  if (diff > 0.30) {
    // 权重差异大 → 系统自动仲裁，高权重方获胜
    // 低权重方收到通知，说明被延迟/取消的原因
    autoDecide(intentA, intentB)

  } else {
    // 权重差异小（≤0.30）→ 上报王奶奶，让她裁决
    // 用通俗易懂的语言描述冲突
    notifyUser(userId, {
      message: "奶奶，今天血压有点高，小助手建议您休息。但是今天原本计划做体操。您想怎么办呀？",
      options: [
        { label: "今天休息，不做体操",  action: "health_first" },
        { label: "做15分钟轻松体操",    action: "compromise"   },
        { label: "正常做体操",          action: "task_first"   },
      ],
      save_as_preference: true  // 王奶奶的选择将被记录，下次类似情况自动应用
    })
  }
}

const writeResult = personalDB.compareAndSet({

  expected_version: 42,        // 我读取时的版本号

  // 获取对方写入的内容
