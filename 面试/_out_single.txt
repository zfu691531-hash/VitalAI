三、拆分方案详解
3.1 拆分后的架构全貌

3.2 组件一：Decision Core（决策核心）——Supervisor 的精华
Decision Core 是拆分后 Supervisor 保留的"灵魂"部分。它只做一件事：接收结构化的事件摘要，输出决策指令。它不关心日志怎么写、不关心Agent心跳、不关心原始数据怎么聚合——这些都交给其他组件。




3.3 组件二：Event Aggregator（事件聚合器）
Event Aggregator 是 Decision Core 的"秘书"。原来所有 Agent 直接"闯进 Supervisor 办公室"汇报，现在必须先经过秘书，秘书整理好、筛选掉重复的，再报给老板。



3.4 组件三：Log Service（日志服务）
日志写入是纯粹的磁盘IO操作，和决策逻辑完全没有关系，但在原来的架构里它和决策竞争同一个 Supervisor 进程的资源。把它独立出来，Decision Core 只需要扔一个"异步日志写入请求"，不等待，继续处理下一个决策。


3.5 组件四：Health Monitor（心跳监控）
Health Monitor 每隔固定时间向所有 Agent 发送心跳探针，如果某个 Agent 连续N次无响应，自动触发告警或重启。这个能力原来隐含在 Supervisor 里，现在独立运行，即使 Decision Core 崩溃，Health Monitor 也能感知并尝试恢复。




四、降级保障机制
拆分之后，Decision Core 崩溃的影响范围已经大幅缩小（其他三个服务还在运行），但如果 Decision Core 长时间不可用，系统需要一套"没有大脑也能运转一段时间"的降级方案。

4.1 Shadow Supervisor（影子Supervisor）——热备接管
Shadow Supervisor 是 Decision Core 的"影子"，它实时接收 Decision Core 同步过来的状态快照。一旦 Health Monitor 检测到 Decision Core 失联，Shadow Supervisor 在5秒内接管所有决策职责。



4.2 各模块自治惯性——最后的保障底线
即使 Decision Core 和 Shadow 都不可用（极端情况），各模块 Agent 应能按"上一次 Supervisor 下发的指令惯性"继续运行至少30分钟，不会立刻瘫痪。这叫"自治惯性"。

拆分是一项有风险的改造，不能一刀切，要渐进执行。以下是三阶段方案，每阶段都可以独立交付、独立验收。

八、总结
Supervisor 的单点问题本质上是"职责过载"的问题，解决方案不是换一个更强的Supervisor，而是把它的职责合理地分配给专职的组件。

// Decision Core 的核心处理逻辑（伪代码）
// 它只关心：收到什么摘要 → 做出什么决策

class DecisionCore {

  // 唯一入口：接收结构化事件摘要
  async processEventSummary(summary) {
    // summary 由 Event Aggregator 预处理好，Decision Core不需要原始数据
    // 例：{ type:"HEALTH_ALERT", priority:1, agent:"agent_health",
    //        content:"血压152/96，连续上升趋势，置信度0.91" }

    const decision = await this.decide(summary)
    // decision 例：{ action:"TRIGGER_P1", target_agents:["agent_daily","agent_mental"],
    //                 instruction:"暂停日常任务，切换健康监控强化模式" }

    // 通过消息队列下发指令（异步，不等待执行结果）
    await msgQueue.dispatch(decision)

    // 只做一件事：同步状态给Shadow Supervisor（热备）
    await shadowSync.push({ summary, decision, timestamp: now() })

    // 日志？不管，扔给Log Service
    logService.writeAsync({ summary, decision })  // fire-and-forget
  }

  // 决策逻辑：根据事件类型 + 当前系统状态 → 输出指令
  async decide(summary) {
    switch(summary.type) {
      case "HEALTH_ALERT":    return this.handleHealthAlert(summary)
      case "TASK_COMPLETE":   return this.handleTaskComplete(summary)
      case "USER_REQUEST":    return this.handleUserRequest(summary)
      case "CONFLICT_REPORT": return this.handleConflict(summary)
      // ... 其他决策分支
    }
  }
}

// 原来的写法（阻塞Decision Core）：
async function handleEvent(event) {
  const decision = await decide(event)
  await db.writeLog(decision)   // ← 等待磁盘写入完成，可能500ms
  await dispatch(decision)      // 决策下发被日志写入延迟了
}

// 优化后的写法（日志服务独立，Decision Core不等待）：
async function handleEvent(event) {
  const decision = await decide(event)
  logService.writeAsync(decision)  // ← fire-and-forget，不等待，约0ms
  await dispatch(decision)          // 决策立即下发，不被日志拖慢
}

// Log Service 内部用消息队列批量写入，效率更高：
class LogService {
  buffer = []
  flushInterval = 500  // 每500ms批量写入一次

  writeAsync(entry) {
    this.buffer.push(entry)  // 仅入队，极快
  }

  flush() {
    if (this.buffer.length > 0) {
      db.batchWrite(this.buffer)  // 批量写入，比单条写入高效10倍
      this.buffer = []
    }
  }
}

// Shadow Supervisor 接管逻辑
class ShadowSupervisor {

  // 实时接收主节点同步的状态（每次决策后推送）
  onStateSync(snapshot) {
    this.latestState = snapshot
    // snapshot包含：当前各Agent状态、最近的决策上下文、待处理事件队列
  }

  // Health Monitor检测到主节点失联后调用
  async takeover() {
    console.log("主节点失联，Shadow接管，状态版本：", this.latestState.version)

    // 1. 从最新快照恢复状态（最多丢失上一次决策后的数据）
    await this.restoreFromSnapshot(this.latestState)

    // 2. 接管Event Aggregator的事件输入
    eventAggregator.redirectTo(this)

    // 3. 通知所有Agent：主节点切换，保持正常运行
    await msgQueue.broadcast({ type:"SUPERVISOR_FAILOVER", new_master:"shadow" })

    // 4. 通知开发者（不告诉用户，降低感知）
    alertDev("Decision Core失联，Shadow已接管，请尽快排查")
  }
}
